## 一、多轮对话响应错位问题

### 1.1 问题场景
用户在连续进行多轮对话时，前端显示的响应总是慢一步，
表现为：
- 第一轮对话正常
- 第二轮对话时，AI的回答内容与第一轮相同
- 第三轮对话时，显示的是第二轮的响应
- 以此类推，每次都显示上一轮的响应

### 1.2 问题分析

#### 1.2.1 初步分析
从后端日志可以看到：
```
Received chat request: requestId=xxx, sessionId=web-xxx, contentLength=5
Found response for session: web-xxx, content length: 32
```
后端返回的响应内容是正确的（32字符），但前端显示的是旧消息（15字符）。

#### 1.2.2 深入分析
通过前端控制台日志发现：
```
[1234567890] Received response: {status: "ok", sessionId: "...", content: "一加一等于二（2）"}
[addMessage] type: bot, content: "一加一等于二（2）"
```
前端收到的响应内容也是正确的，但显示的内容不对。

#### 1.2.3 根因定位
最终定位到问题出在消息总线的响应路由机制：

1. **并发竞争条件**：多个请求并发发送到消息总线
2. **响应匹配问题**：消息总线使用单一的出站队列（`BlockingQueue<OutboundMessage>`）
3. **会话ID匹配不精确**：虽然使用会话ID（chatId）来匹配响应，但由于并发竞争，响应被错误的请求者消费
4. **消息缓存问题**：旧的响应消息没有被正确清理，导致被重复消费

#### 1.2.4 时序问题
```
请求1 (chatId=A) ──────────────────────────────────────────────────────▶ 响应1 (chatId=A)
请求2 (chatId=A) ─────────────────▶ 响应2 (chatId=A) ─────────────────▶ 显示错误
```
请求2 在响应1到达之前就消费了响应1，导致显示错误的响应。

### 1.3 解决方案

#### 1.3.1 架构改进
采用**精确匹配**机制，使用请求ID（requestId）来确保每个请求收到正确的响应。

#### 1.3.2 具体实现

**步骤1：在 OutboundMessage 中添加 requestId 字段**

```java
// OutboundMessage.java
public class OutboundMessage {
    private final String requestId;  // 新增字段
    
    public Builder requestId(String requestId) {
        this.requestId = requestId;
        return this;
    }
}
```

**步骤2：修改 MessageBus，添加会话响应映射**

```java
// MessageBus.java
private final ConcurrentHashMap<String, Queue<OutboundMessage>> sessionResponses;

// 发布响应时，同时添加到会话响应映射
public void publishOutbound(OutboundMessage message) {
    outboundQueue.put(message);
    if (message.getChatId() != null) {
        sessionResponses.computeIfAbsent(message.getChatId(), k -> new LinkedList<>())
                       .offer(message);
    }
}

// 使用 requestId 精确匹配响应
public OutboundMessage waitForSessionResponse(String sessionId, String requestId, 
                                             long timeout, TimeUnit unit) {
    long endTime = System.currentTimeMillis() + unit.toMillis(timeout);
    
    while (System.currentTimeMillis() < endTime) {
        Queue<OutboundMessage> queue = sessionResponses.get(sessionId);
        if (queue != null) {
            for (OutboundMessage msg : queue) {
                if (requestId != null && requestId.equals(msg.getRequestId())) {
                    queue.remove(msg);
                    return msg;
                }
            }
        }
        Thread.sleep(50);
    }
    return null;
}
```

**步骤3：修改 ChannelServer，传递 requestId**

```java
// ChannelServer.java
// 从请求中获取 requestId
String requestId = json.has("requestId") ? json.get("requestId").asText() : UUID.randomUUID().toString();

// 等待响应时使用 requestId 精确匹配
String responseContent = waitForResponse(sessionId, requestId, 60);
```

**步骤4：修改 AgentLoop，传递 requestId**

```java
// AgentLoop.java
private TurnState doRespond(TurnContext context) {
    String requestId = null;
    if (context.getMessage().getMetadata() != null) {
        Object requestIdObj = context.getMessage().getMetadata().get("requestId");
        if (requestIdObj != null) {
            requestId = requestIdObj.toString();
        }
    }
    
    OutboundMessage response = OutboundMessage.builder()
        .channel(context.getMessage().getChannel())
        .chatId(context.getMessage().getChatId())
        .content(content)
        .requestId(requestId)  // 传递 requestId
        .build();
    
    messageBus.publishOutbound(response);
}
```

**步骤5：前端添加请求队列**

```javascript
// index.html
let requestQueue = [];
let isProcessing = false;

function processQueue() {
    if (isProcessing || requestQueue.length === 0) return;
    isProcessing = true;
    const task = requestQueue.shift();
    try {
        await task();
    } finally {
        isProcessing = false;
        processQueue();
    }
}

function sendMessage() {
    // ... 添加用户消息 ...
    addToQueue(async () => {
        await sendMessageHTTP(content);
    });
}
```

#### 1.3.3 优化后的时序
```
请求1 (requestId=req1) ──────────────────────▶ 响应1 (requestId=req1) ─────────────────▶ 显示正确
请求2 (requestId=req2) ─────────────────▶ 响应2 (requestId=req2) ─────────────────▶ 显示正确
```

## 二、流式对话第2轮起无响应/超时

### 2.1 问题场景
用户通过 SSE 流式对话接口 (`/api/chat/stream`) 进行多轮对话：
- **第1轮**：正常返回流式内容 ✅
- **第2轮**：偶尔正常，经常超时无内容 ❌
- **第3轮及以后**：后端完全没有反应（SSE 连接建立后立即超时，`dataCount=0`）❌

关键日志特征：
```
SSE timeout for request: xxx (dataCount=0)   ← LLM从未开始流式输出
AsyncRequestTimeoutException                  ← Spring MVC 30秒异步超时
```

### 2.2 问题分析

#### 2.2.1 第一层根因：Spring MVC 异步超时 30 秒

Spring Boot 默认 `spring.mvc.async.request-timeout` 为 **30秒**。当 LLM 调用（DeepSeek API）超过30秒时，Spring 强制关闭 SSE 连接并抛出 `AsyncRequestTimeoutException`。

`SseEmitter` 自身设了 `SSE_TIMEOUT=300000`（5分钟），但被 Spring MVC 层提前截断。

#### 2.2.2 第二层根因：僵尸 callback 堆积

关键时刻 `emitter.onTimeout` 只打日志，**没有清理 callback**：
```java
// 修复前
emitter.onTimeout(() -> {
    logger.warn("SSE timeout");  // 只打日志！callback 永远留在列表里
});
```

而 `emitter.onCompletion` 正常时会清理。超时路径不走 `onCompletion` → callback 泄漏 → 僵尸对象堆积在 `CopyOnWriteArrayList` 中，后续请求遍历列表时调用已关闭的 `emitter.send()` 抛 `IllegalStateException`。

#### 2.2.3 第三层根因：AgentLoop 单线程串行阻塞

`AgentLoop.runLoop()` 是单线程串行循环：
```java
// 修复前
while (running.get()) {
    InboundMessage message = messageBus.consumeInbound(1, TimeUnit.SECONDS);
    if (message != null) {
        processMessage(message);  // ← 同步阻塞！LLM调用不返回，loop就卡死
    }
}
```

当某条消息的 `doRun()` → `runner.run().join()` 中 LLM 调用卡住（DeepSeek 响应慢/超时重试）时：
- AgentLoop 线程被阻塞，无法消费新消息
- 新消息在 `ArrayBlockingQueue` 中排队
- `ChatController.publishInbound()` 成功（队列未满，容量100）
- 返回 `SseEmitter` 给前端
- 但 `doRun()` 从未执行 → `dataCount=0` → SSE 超时

#### 2.2.4 第四层根因：两个 AgentLoop 竞争

Spring Boot 启动了**两个** `AgentLoop` 实例共享同一个 `MessageBus`：

| AgentLoop | 来源 | callback 设置 |
|-----------|------|--------------|
| Spring Bean | `NanobotConfig.agentLoop()` | WebSocket `initStreamCallback` |
| NanobotRunner | `NanobotRunner.run()` 创建 | ChatController SSE callback |

两者竞争消费 `MessageBus` 中的消息。ChatController 把 callback 设在 NanobotRunner 的 AgentLoop 上，但消息可能被 Spring Bean 抢走 → callback 不匹配 → 前端收不到数据。

### 2.3 解决方案

#### 修复1：延长 Spring MVC 异步超时

```yaml
# application.yml — 合并到已有 spring: 块中（修复了重复 key 导致覆盖的 bug）
spring:
  mvc:
    async:
      request-timeout: 300000   # 5分钟，匹配 SSE_TIMEOUT
```

#### 修复2：超时也清理 callback + 扩大异常捕获

```java
// ChatController.java — onTimeout 也清理
emitter.onTimeout(() -> {
    agentLoop.removeStreamResponseCallback(callback);  // ← 新增
});

// 扩大 catch 范围（原来只 catch IOException，遗漏了 IllegalStateException）
try {
    emitter.send(SseEmitter.event().data(content));
} catch (Exception e) {  // ← 原来只 catch IOException
    // ...
}
```

#### 修复3：AgentLoop 异步消息处理

```java
// AgentLoop.java — runLoop() 不再同步阻塞
while (running.get()) {
    InboundMessage message = messageBus.consumeInbound(1, TimeUnit.SECONDS);
    if (message != null) {
        // 提交到线程池异步处理，主循环立即返回继续消费
        messageExecutor.submit(() -> processMessage(message));
    }
}
```

#### 修复4：消除双 AgentLoop

`NanobotRunner` 不再自行创建 `AgentLoop`/`MessageBus`/`ToolRegistry`/`SessionManager`，统一通过 `@Autowired` 注入 Spring Bean。

#### 修复5：单回调 → 多回调列表

`AgentLoop.streamResponseCallback` 从单个字段改为 `CopyOnWriteArrayList<StreamResponseCallback>`，支持 SSE + WebSocket 同时注册回调。
