# 数据模型与存储 Schema

> **作用域**: mynanobot-java 的数据模型字段定义与存储 Schema。
> **重要约束（ADR-006）**: 用户 PII 不以原文落盘。Redis（短期 TTL）+ 内存 + ReMe 向量库（长期记忆）为主存储；MySQL `huazai_trip_trace` 库存储脱敏后的审计 Trace（ADR-010）和历史规划结果（ADR-012）。

---

## 1. 存储分层（ADR-006）

| 层 | 介质 | 内容 | 生命周期 |
|----|------|------|---------|
| 短期状态 | Redis 7.x | 会话上下文、Agent 中间结果、用户偏好、笔记缓存 | TTL 24h |
| 长期记忆 | AgentScope ReMe（向量库） | 历史行程、用户画像、热门景点知识 | 永久（策略淘汰） |
| 审计追踪 | MySQL `huazai_trip_trace` | Agent 委派/LLM 调用/编排汇总（ADR-010） | 永久（建议 90d） |
| 历史规划 | MySQL `huazai_trip_trace` | 脱敏后规划结果（ADR-012）：行程/景点/餐饮/路线/费用 | 永久（建议 180d） |
| 临时文件 | 沙箱 `/tmp/agentscope/{agentId}/` | PDF、图片缓存 | 任务结束清理 |

## 2. Java 领域模型字段定义

> 实体均位于 `huazai-trip-common` 的 `model/` / `dto/`。下表为字段级契约。

### 2.1 TripPlanRequest（dto）
| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| destination | String | 非空 | 目的地 |
| startDate | LocalDate | 非空 | 起始日期 |
| days | int | >0, ≤30 | 天数 |
| budget | BigDecimal | >0 | 总预算（元） |
| headcount | int | ≥1 | 人数 |
| travelStyle | TravelStyle | 枚举 | FAMILY/COUPLE/SOLO/TEAM |
| preferences | List\<String> | 可空 | 偏好标签 |

### 2.2 TripPlan（model）
| 字段 | 类型 | 说明 |
|------|------|------|
| planId | String | 主键，格式 `P-yyyyMMdd-NNN` |
| request | TripPlanRequest | 原始请求 |
| days | List\<TripDay> | 每日行程 |
| totalCost | BigDecimal | 总费用 |
| budget | Budget | 预算与拆分 |
| status | PlanStatus | 见状态枚举 |
| requiresHITL | boolean | 是否需人工介入 |

### 2.3 TripDay（model）
| 字段 | 类型 | 说明 |
|------|------|------|
| date | LocalDate | 当日日期 |
| dayIndex | int | 第几天（1-based） |
| attractions | List\<Attraction> | 景点（有序） |
| meals | List\<Meal> | 餐饮 |
| routes | List\<Route> | 点间路线 |
| dailyCost | BigDecimal | 当日小计 |

### 2.4 Attraction / Meal / Route / Budget / GeoLocation
| 实体 | 关键字段 |
|------|---------|
| Attraction | attractionId, name, location:GeoLocation, rating(≥3.5), durationMin, ticketPrice, tags |
| Meal | name, location:GeoLocation, mealType(BREAKFAST/LUNCH/DINNER), avgPrice, recommendDishes |
| Route | from:GeoLocation, to:GeoLocation, distanceKm, durationMin(≤120), transportMode, fallback:boolean |
| Budget | total, used, overrunRate, breakdown:{transport,hotel,food,ticket} |
| GeoLocation | longitude, latitude, address, city |

### 2.5 XHSNote（common 模块 · C-006 已落地，增量 B 已改情感粒度）
| 字段 | 类型 | 说明 |
|------|------|------|
| noteId | String | 笔记 ID |
| title / content | String | 标题/正文 |
| tags | List\<String> | 标签 |
| likeCount | int | 点赞 |
| mentions | List\<AttractionMention> | 景点提及（**逐提及情感**，取原文时为空，由 `XHSNoteParser` 回填） |

**AttractionMention（common 模块 · C-006 增量 B 新增 · 不可变）**
| 字段 | 类型 | 说明 |
|------|------|------|
| attractionId | String | 提及景点（**ID**，非名称；链接到 `AttractionCatalog` 的键） |
| sentiment | SentimentType | POSITIVE/NEUTRAL/NEGATIVE（**逐提及单值**） |

> **落地说明（C-006）**：`XHSNote` 实际落于 `huazai-trip-common`（与 `Attraction`/`SentimentType` 同层；构造器对 `tags`/`mentions` 做 null 安全不可变拷贝）。`XHSNoteAnalyzer` 按 `mentions` 的 `attractionId` 聚合，对每个提及计入其自身 `sentiment`，再做评分(≥3.5)/情感筛选 + 频次排序——「一篇夸 A 吐槽 B」可分别正确计入。候选 `Attraction` 的真实 rating/名称/位置经 `AttractionCatalog` 端口解析（主链路桩，真实 ReMe `attraction_kb` 召回延后 C-015）。
>
> **情感粒度契约变更（C-006 增量 B / AC-11，已落地 §2.5）**：情感由整篇单值下沉到每个提及——新增不可变 `AttractionMention{attractionId, sentiment}`，以 `mentions: List<AttractionMention>` 取代原 `mentionedAttractions: List<String>` + 整篇 `sentiment: SentimentType`（字段级破坏性变更，A2A `TASK_RESULT` 信封不变）。原拟 **C-017 占位作废**，D-1/D-2/D-3 收编进 C-006 增量 B。
> **解析端口（C-006 增量 B / AC-10）**：「原始笔记→结构化」由独立 `XHSNoteParser` 端口承载（LLM 抽取/链接 `AttractionCatalog`/逐提及情感，经 `GovernedExternalCaller` 治理）；`XHSNoteSource` 退化为只取原文。本步已落端口抽象与契约迁移，LLM 真实实现 + 串联编排 + 官方 API 来源适配为增量 B 后续步骤。

## 4. Redis Key Schema

> 统一前缀 `trip:`，Key 常量集中于 `CacheConstants`。所有 Key 默认 TTL 24h。

| Key 模式 | 类型 | 内容 | TTL  |
|---------|------|------|------|
| `trip:plan:{planId}` | Hash | TripPlan 序列化（JSON 分字段） | 30d  |
| `trip:plan:{planId}:status` | String | 当前 PlanStatus | 30d  |
| `trip:session:{sessionId}` | Hash | 会话上下文（AgentScope Session） | 24h  |
| `trip:step:{planId}` | Hash | PlanNotebook 中间步骤（断点续传） | 24h  |
| `trip:pref:{userId}` | Hash | 用户偏好快照 | 24h  |
| `trip:xhs:note:{noteId}` | String | 小红书笔记缓存（JSON） | 24h  |
| `trip:ratelimit:{api}:{window}` | String | 限频计数 | 窗口长度 |
| `trip:agent:result:{planId}:{agent}` | String | 子 Agent 产出物缓存 | 24h  |
| `trip:plan:seq:{yyyyMMdd}` | String | planId 每日序列号（Redis INCR，ADR-012） | 48h |
| `trip:plan:{planId}:draft` | String | 计划草案 Markdown（C-018 AC-3） | 30d |

**约定**
- 值序列化用 `JsonUtils`（Jackson），时间字段 ISO-8601。
- 大对象按字段拆 Hash，避免单 Key 过大。
- 限频用 `INCR` + `EXPIRE` 实现滑动窗口。

## 5. ReMe 长期记忆 Schema（ADR-011，C-015 落地）

> **存储介质**: ReMe Server（Python，`reme-ai[core]`，Docker 部署）。Java 侧通过 `ReMeClient`（`agentscope-extensions-reme`）读写。
> **隔离模型**: workspace = userId（用户维度隔离）；共享知识使用固定 workspace `attraction_kb`。
> **Embedding**: ReMe Server 内置（DashScope `text-embedding-v3`），Java 侧不直接调用。

| workspace | 写入内容 | 写入时机 | 召回用途 |
|-----------|---------|---------|---------|
| `{userId}` | 行程摘要（planId + 目的地/风格/天数/费用/关键景点） | confirmed 后异步写入 | 二次规划同目的地/风格时召回历史行程 |
| `{userId}` | 用户画像（travelStyle + preferences 标签） | 规划请求时累积更新 | 召回偏好标签增强推荐 |
| `attraction_kb` | 景点知识（名称/评级/标签/Tips，来自 XHS 分析产出） | XHS 分析后沉淀 | 召回目的地相关景点知识 |

**Java 侧模型**:
- `MemoryContext`（`huazai-trip-common/dto`）: record，含 `userProfiles`/`historyPlans`/`attractionKb`（均 `List<String>`）+ `memoryFallback`（boolean）。常量 `EMPTY`（空记忆）/ `FALLBACK`（降级标记）。
- `MemoryRecallService`（`huazai-trip-skills/memory`）: 召回三类记忆，封装为 `MemoryContext`。降级返回 `FALLBACK`。
- `MemoryWriteService`（`huazai-trip-skills/memory`）: 三类记忆异步写入 ReMe。
- `MemoryClearService`（`huazai-trip-skills/memory`）: HTTP DELETE 清除用户 workspace。

**召回策略**: 注入 `TripOrchestrationQuery.memoryContext`，经 Supervisor 传递至 XHS/Itinerary Agent 上下文。记忆仅作弱增强（prompt context），不覆盖 `TripPlanRequest` 硬约束。

**降级**: ReMe 不可用 → `MemoryContext.FALLBACK`（`memoryFallback=true`），主链路照常规划。

**数据清除**: `DELETE /api/v1/trip-plan/memory` → `MemoryClearService.clearWorkspace(userId)` → ReMe HTTP DELETE `/workspace/{userId}`。

**可观测性**: `memory.recall.latency`（Timer）、`memory.recall.hit`（Counter, tag=hit/miss）、`memory.fallback.count`（Counter）。

## 6. MySQL 业务表 Schema（ADR-012，C-017）

> **数据库**: `huazai_trip_trace`（复用 ADR-010 审计库）。6 张表通过 `plan_id` 关联。
> **脱敏**: 落库前 `SensitiveDataMasker` 处理 API Key + 手机号。TEXT 字段截断至 `MAX_TEXT_LENGTH`。
> **写入**: 异步（`TraceThreadPoolConfig` 线程池），失败仅 WARN + metrics，不阻塞主链路。

| 表名 | 主键 | 关联 | 说明 |
|------|------|------|------|
| `trip_plan_record` | `id` BIGINT | `plan_id` VARCHAR(64) | 规划主表（1:1 TripPlan，含 destination/status/费用/人数/风格） |
| `trip_day_record` | `id` BIGINT | `plan_id` + `day_index` | 每日行程（1:N TripDay，含 date/dailyCost） |
| `trip_attraction_record` | `id` BIGINT | `day_id` FK | 景点（1:N Attraction，含名称/评分/门票/时长/经纬度） |
| `trip_meal_record` | `id` BIGINT | `day_id` FK | 餐饮（1:N Meal，含类型/餐厅/人均） |
| `trip_route_record` | `id` BIGINT | `day_id` FK | 路线（1:N Route，含起终点/距离/耗时/交通方式/fallback） |
| `trip_budget_record` | `id` BIGINT | `plan_id` | 费用明细（1:1 Budget，含总额/交通/住宿/餐饮/门票/超支率） |

**索引**: `plan_id`（所有表）、`user_id`（plan_record）、`created_at`（plan_record）、`destination`（plan_record）。

**软删除**: `trip_plan_record.deleted` TINYINT(1) DEFAULT 0，列表查询自动过滤。

## 7. 数据安全与合规

| 规则 | 说明 |
|------|------|
| 不落盘 | 用户数据仅 Redis + 内存，不写本地磁盘/关系库 |
| 脱敏 | 手机号、身份证等在存储与日志中脱敏 |
| 沙箱 | 临时文件限定 `/tmp/agentscope/{agentId}/`，≤10MB |
| 自动过期 | Redis 24h TTL，无需手动清理 |
| 密钥隔离 | API Key 走环境变量/配置中心，不入任何数据模型 |

> 业务规则见 `业务模型.md`；接口字段见 `接口协议.md`。
