# 接口协议定义

> **作用域**: mynanobot-java 对外 REST API 与内部 A2A / MCP 协议约定。
> **定位**: 接口契约真相来源。设计与联调时按需加载。
> **版本**: v1

---

## 1. 对外 REST API

基础路径: `/api/v1` · 媒体类型: `application/json` · 字符集: UTF-8

### 1.1 提交规划请求

```
POST /api/v1/trip-plan
```

**Request**
```json
{
  "destination": "云南",
  "startDate": "2026-07-01",
  "days": 5,
  "budget": 8000,
  "headcount": 2,
  "travelStyle": "COUPLE",
  "preferences": ["古镇", "美食", "小众"]
}
```

**Response 202 Accepted**（异步规划，返回 planId 轮询）
```json
{ "planId": "P-20260701-001", "status": "analyzing" }
```

**错误码**
| HTTP | code | 含义 |
|------|------|------|
| 400 | INVALID_REQUEST | 参数缺失/非法（如 days ≤ 0） |
| 429 | RATE_LIMITED | 触发限频 |
| 503 | SERVICE_UNAVAILABLE | 下游 Agent/外部 API 不可用 |

**验收示例**
- 当请求参数合法时，返回 `202 + planId + status=analyzing`
- 当 `days <= 0` 时，返回 `400 INVALID_REQUEST`
- 当下游依赖不可用时，返回 `503 SERVICE_UNAVAILABLE`，并带 `traceId`

### 1.2 查询规划进度/结果

```
GET /api/v1/trip-plan/{planId}
```

**Response 200**
```json
{
  "planId": "P-20260701-001",
  "status": "review",
  "plan": {
    "days": [ { "date": "2026-07-01", "attractions": [ ... ], "meals": [ ... ], "dailyCost": 1450 } ],
    "totalCost": 7820,
    "budget": { "total": 8000, "overrunRate": -0.02, "breakdown": { "transport": 2200, "hotel": 3000, "food": 1620, "ticket": 1000 } }
  },
  "requiresHumanIntervention": false
}
```

`status`: `analyzing | planning | plan_review | assembling | budgeting | review | confirmed | failed`

**状态语义**
- `analyzing`: XHS / 初始理解阶段
- `planning`: 路线与任务分解阶段
- `assembling`: 行程编排阶段
- `budgeting`: 预算核算阶段
- `review`: 已生成草案，等待 HITL 或最终确认
- `confirmed`: 已确认可交付
- `failed`: 主链路失败，需重试或人工排查

### 1.3 人工介入（HITL 确认/修改）

```
POST /api/v1/trip-plan/{planId}/intervene
```
```json
{ "action": "MODIFY", "dayIndex": 2, "changes": { "removeAttraction": "A-007", "addAttraction": "A-012" } }
```
`action`: `CONFIRM | MODIFY | REJECT`（plan_review 态不支持 MODIFY，C-018 AC-4）

**验收示例**
- `CONFIRM`: 当前草案进入确认态
- `MODIFY`: 当前草案根据变更请求重算或局部调整
- `REJECT`: 当前方案作废，回到重新规划或失败处理

### 1.4 用户反馈

```
POST /api/v1/trip-plan/{planId}/feedback
```
```json
{ "rating": "UP", "modifiedItems": ["day-2-attraction"], "comment": "整体满意" }
```

### 1.5 导出行程方案 PDF（C-014）

```
GET /api/v1/trip-plan/{planId}/export?format=pdf
```

**需认证**: 携带 `Authorization: Bearer <accessToken>`。

**Response 200 OK** — `Content-Type: application/pdf`, `Content-Disposition: attachment; filename="{planId}.pdf"`

PDF 内容（AC-2）:
- 封面：目的地、出发日期、天数、人数、旅行风格、方案编号、总费用
- 逐日行程：景点（名称/时长/门票/评分）、餐饮（类型/餐厅/人均）、路线（起点→终点/距离/耗时/交通方式）
- 费用拆分：交通、住宿、餐饮、门票 + 总预算/实际费用/超支率
- 页脚：traceId、生成时间、planId

**状态门禁**: 仅 `status ∈ {review, confirmed}` 可导出（AC-1）。
- `review` → PDF 带「草案·未确认」水印（AC-3）
- `confirmed` → 正式版 PDF，无水印

**字段安全**: traceId 在 PDF 页脚脱敏显示（仅首尾各 4 位）；敏感个人字段不进入 PDF。

**错误码**
| HTTP | code | 含义 |
|------|------|------|
| 400 | PLAN_NOT_EXPORTABLE | 方案当前状态不可导出（仅 review/confirmed 可导出） |
| 400 | INVALID_REQUEST | format 参数不合法（目前仅支持 pdf） |
| 404 | NOT_FOUND | planId 不存在或行程数据不存在 |
| 403 | FORBIDDEN | 非本人 plan（userId 不匹配） |

**验收示例**
- 输入 `GET .../P-20260701-001/export?format=pdf`（status=confirmed） → `200 OK`，PDF 含封面+逐日+费用+页脚，无水印。
- 输入 `GET .../P-NOT-EXIST/export?format=pdf` → `404 { code: "NOT_FOUND", message: "...", traceId: "..." }`。
- 输入 `GET .../P-001/export?format=pdf`（status=analyzing） → `400 { code: "PLAN_NOT_EXPORTABLE", message: "...", traceId: "..." }`。
- 输入 `GET .../P-001/export?format=pdf`（status=review） → `200 OK`，PDF 带「DRAFT - 未确认」水印。
- 输入 `GET .../P-001/export?format=pdf`（含 fallback=true 路线） → PDF 对应条目标注 fallback 标记。
- 输入 `GET .../P-001/export?format=xlsx` → `400 { code: "INVALID_REQUEST", message: "不支持的导出格式: xlsx" }`。

### 1.6 查询历史规划列表（C-017 / AC-3）

```
GET /api/v1/trip-plan/history?page=1&pageSize=10&keyword=北京&travelStyle=FAMILY&planStatus=confirmed&startDate=2026-01-01&endDate=2026-12-31
```

**需认证**: 携带 `Authorization: Bearer <accessToken>`。userId 从 SecurityContext 提取。

**Query 参数**
| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| page | int | 否（默认 1） | 页码（1-based） |
| pageSize | int | 否（默认 10，最大 50） | 每页条数 |
| keyword | String | 否 | 目的地模糊搜索（原有参数） |
| travelStyle | String | 否 | 旅行风格筛选（FAMILY/COUPLE/SOLO/TEAM） |
| planStatus | String | 否 | 状态筛选（confirmed/failed/review 等） |
| startDate | String | 否 | 起始日期（ISO-8601 yyyy-MM-dd） |
| endDate | String | 否 | 结束日期（ISO-8601 yyyy-MM-dd） |

**Response 200 OK**
```json
{
  "items": [
    { "planId": "P-20260701-001", "destination": "北京", "startDate": "2026-07-01", "days": 3, "budget": 8000, "totalCost": 7820, "headcount": 2, "travelStyle": "FAMILY", "status": "confirmed", "createdAt": 1719792000000 }
  ],
  "total": 1,
  "page": 1,
  "pageSize": 10
}
```

**查询策略**: Redis-first（PlanCacheService ZSet + PlanSummary 缓存，30 天 TTL）→ MySQL-fallback（超出 TTL 后降级查询 `trip_plan_record`）。

### 1.7 查询历史规划详情（C-017 / AC-4）

```
GET /api/v1/trip-plan/history/{planId}
```

**需认证**: 携带 `Authorization: Bearer <accessToken>`。

**Response 200 OK**
```json
{
  "planId": "P-20260701-001",
  "destination": "北京",
  "status": "confirmed",
  "plan": { "days": [...], "totalCost": 7820, "budget": {...} },
  "traceSummary": { "delegationCount": 4, "totalDurationMs": 25000, "fallbackCount": 0, "hitlTriggered": false }
}
```

**错误码**
| HTTP | code | 含义 |
|------|------|------|
| 404 | NOT_FOUND | planId 不存在或已软删除 |
| 403 | FORBIDDEN | 非本人 plan |

### 1.8 软删除历史规划（C-017 / AC-5）

```
DELETE /api/v1/trip-plan/history/{planId}
```

**需认证**: 携带 `Authorization: Bearer <accessToken>`。

**Response 200 OK**
```json
{ "message": "已删除" }
```

逻辑删除（`deleted=1`），列表查询自动过滤；详情接口返回 404。

### 1.9 清除用户记忆（C-015 / AC-10）

```
DELETE /api/v1/trip-plan/memory
```

**需认证**: 携带 `Authorization: Bearer <accessToken>`。userId 从 SecurityContext 提取，非 URL 参数。

**Response 200 OK**
```json
{ "success": true }
```

**Response 200 OK（ReMe 不可用时降级）**
```json
{ "success": false }
```

清除当前认证用户在 ReMe Server 中的全部长期记忆（workspace = userId）。降级场景返回 `success=false`，不抛异常。

**用途**: 合规要求，用户有权随时清除个人数据（ADR-011）。

### 1.11 认证端点（C-011 AC-1/AC-2/AC-3）

> 注册/登录端点公开（无需 token）；登出/刷新需携带有效 token。

#### POST /api/v1/auth/register

**Request**
```json
{ "username": "test", "password": "Pass1234", "nickname": "测试" }
```
**Response 201 Created**
```json
{ "userId": 1, "username": "test" }
```
**错误码**
| HTTP | code | 含义 |
|------|------|------|
| 400 | INVALID_REQUEST | 密码不满足格式要求（长度 < 8） |
| 409 | CONFLICT | username 已存在 |

---

#### POST /api/v1/auth/login

**Request**
```json
{ "username": "test", "password": "Pass1234" }
```
**Response 200 OK**
```json
{ "accessToken": "...", "refreshToken": "...", "expiresIn": 7200 }
```
同时写入 Redis：`auth:access:{userId}`（TTL 2h）、`auth:refresh:{userId}`（TTL 7d）。

**错误码**
| HTTP | code | 含义 |
|------|------|------|
| 401 | UNAUTHORIZED | 用户名或密码错误（防枚举，不区分） |

---

#### POST /api/v1/auth/logout

需携带有效 `Authorization: Bearer <accessToken>`。

**Response 200 OK**
```json
{ "message": "已登出" }
```
同时删除 Redis 中 `auth:access:{userId}` + `auth:refresh:{userId}`（旧 token 即吊销）。

---

#### POST /api/v1/auth/refresh

**Request**
```json
{ "refreshToken": "..." }
```
**Response 200 OK**
```json
{ "accessToken": "...", "expiresIn": 7200 }
```
同时更新 Redis 中 `auth:access:{userId}`。

**错误码**
| HTTP | code | 含义 |
|------|------|------|
| 401 | UNAUTHORIZED | refreshToken 无效/过期/不匹配 |

## 2. 内部 A2A 协议（Agent-to-Agent）

> AgentScope 原生 A2A + Nacos 注册发现。消息为不可变 `Msg`。

### 2.1 委派任务（Supervisor → SubAgent）
```json
{
  "msgId": "M-...",
  "from": "supervisor",
  "to": "route-agent",
  "type": "TASK_ASSIGN",
  "payload": { "taskType": "ROUTE_PLANNING", "context": { "destination": "云南", "attractions": ["A-001", "A-002"] } },
  "timeoutMs": 30000,
  "traceId": "T-..."
}
```

### 2.2 返回结果（SubAgent → Supervisor）
```json
{
  "msgId": "M-...",
  "from": "route-agent",
  "to": "supervisor",
  "type": "TASK_RESULT",
  "payload": { "status": "OK", "result": { ... }, "fallback": false },
  "traceId": "T-..."
}
```

**消息类型**: `TASK_ASSIGN | TASK_RESULT | TASK_ERROR | HEARTBEAT | BROADCAST`

### 2.3 契约演进规则
- A2A `payload` 默认采用向后兼容演进，允许新增字段，禁止无版本说明地删除字段或改变既有字段语义。
- 破坏性变更必须通过 ADR 或变更卡显式说明，并提供过渡策略。
- `fallback`、`traceId`、`timeoutMs` 视为协同链路关键字段，不得在新版本中静默移除。
- 子 Agent 若无法识别新增字段，应忽略而非失败，除非该字段被声明为必填。

**调用约束**
| 约束 | 值 |
|------|-----|
| 单次超时 | 30s |
| 重试 | 3 次（指数退避 1/2/4s） |
| 调用链深度 | ≤ 5（防死锁） |
| 失败降级 | 返回默认值 + 标记 `fallback: true` |

**测试关注点**
- `TASK_ASSIGN` 必须包含 `traceId / timeoutMs / payload`
- `TASK_RESULT` 在降级场景下必须显式带 `fallback=true`
- 子 Agent 失败时必须返回可识别错误，而不是静默超时

百度/高德地图作为外部 MCP Server 接入，经适配层封装为 Tool。

| MCP Tool | 用途 | 限频 | 降级 |
|----------|------|------|------|
| `baidu.route` | 路线规划 | ≤30/min | 离线直线距离估算 |

| `baidu.traffic` | 实时路况 | ≤30/min | 忽略路况，用平均速度 |

## 5. 通用约定

- 所有响应含 `traceId`，贯穿 OpenTelemetry 链路。
- 时间用 ISO-8601（`yyyy-MM-dd` / `yyyy-MM-ddTHH:mm:ssZ`）。
- 金额单位为人民币元，整数或两位小数。
- 错误响应统一结构: `{ "code": "...", "message": "...", "traceId": "..." }`。
- 敏感字段（用户手机号等）在日志与响应中脱敏。

> 字段级数据定义见 `.harness/wiki/数据模型.md`；业务规则见 `业务模型.md`。

## 6. 可观测性指标与告警

| 指标 | 说明 | 告警阈值 |
|------|------|---------|
| `agent.call.latency` | Agent 调用延迟（P95） | > 10s |
| `agent.call.error.rate` | Agent 调用错误率 | > 5% |
| `llm.token.usage` | LLM Token 用量 | 按预算告警 |
| `api.rate_limit.hit` | API 限频命中次数 | > 0 |
| `memory.cache.hit.rate` | 记忆缓存命中率 | < 80% |
| `memory.recall.latency` | ReMe 记忆召回延迟（Timer，P95 < 1s） | > 1s |
| `memory.recall.hit` | 记忆召回命中/未命中计数（tag: result=hit/miss） | — |
| `memory.fallback.count` | 记忆降级次数（ReMe 不可用） | > 0 |
| `human.intervention.count` | HITL 触发次数 | 按 plan 异常增多 |
| `plan.notebook.steps` | PlanNotebook 步数分布 | > 15 步（可能死循环） |
| `trip.persistence.write.success` | 业务持久化写入成功（tag: table） | — |
| `trip.persistence.write.failure` | 业务持久化写入失败（tag: table） | > 0 |

每个 Agent 暴露 `GET /actuator/health` + `GET /actuator/metrics` + `GET /actuator/info`。指标暴露框架为 Micrometer → Prometheus，链路追踪为 OpenTelemetry。
