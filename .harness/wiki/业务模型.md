# 业务模型与实体关系

> **作用域**: mynanobot-java 旅游规划领域的核心业务概念、实体、关系。
> **定位**: 领域知识真相来源。需求分析与设计时按需加载。

---

## 1. 业务全景

小红书旅游规划多智能体系统：用户给出旅游意图（目的地/天数/预算/偏好），5 个 Agent 自主协作产出一份完整、合理、个性化、不超预算的行程方案。

```
用户意图 ──▶ Supervisor（任务分解）
                ├─▶ XHS（小红书笔记分析，挖掘热门玩法）
                ├─▶ Route（路线规划，地图 MCP）
                ├─▶ Itinerary（行程编排，景点/餐饮/时间槽）
                └─▶ Budget（费用统筹，超支预警）
                        ▼
                  完整 TripPlan ──▶ 用户（HITL 确认）
```

### 1.1 范围边界

| 包含（In Scope） | 不包含（Out of Scope） |
|-----------------|----------------------|
| ✅ 小红书笔记内容检索与分析 | ❌ 小红书内容发布/评论 |
| ✅ 基于笔记的景点推荐 | ❌ 机票/酒店实时预订 |
| ✅ 多日行程自动编排 | ❌ 社交功能/用户系统 |
| ✅ 交通路线规划（百度/高德地图） | ❌ 实时导航 |
| ✅ 费用预算计算与管控 | ❌ 在线支付 |
| ✅ 行程方案 PDF 导出 | ❌ 移动端 App |

### 1.2 质量承诺
- 核心 Agent 逻辑测试覆盖率 ≥ 80%
- 所有外部 API 调用有降级方案
- 端到端协作流程可手动复现
- 文档与代码同步更新

### 1.3 合规声明
- 仅用于学习与个人使用，不做商业化部署
- 遵守小红书 robots.txt 与 API 使用条款
- 小红书数据仅用于行程规划参考，缓存 24h
- 用户有权随时清除个人数据（`DELETE /api/v1/trip-plan/memory` 清除 ReMe 长期记忆）；API Key 轮换周期 90 天
- 长期记忆以 workspace = userId 隔离，不跨用户泄漏（ADR-011）

## 2. 核心业务实体

| 实体 | 含义 | 关键属性 |
|------|------|---------|
| **TripPlanRequest** | 用户的规划请求 | 目的地、起止日期、天数、预算、人数、旅行风格、偏好标签 |
| **TripPlan** | 完整行程方案（最终交付物） | 请求引用、TripDay 列表、总费用、Budget、状态 |
| **TripDay** | 单日行程 | 日期、序号、Attraction 列表、Meal 列表、Route 列表、当日小计 |
| **Attraction** | 景点 | 名称、GeoLocation、评分、建议游玩时长、门票、标签 |
| **Meal** | 餐饮安排 | 餐厅名、GeoLocation、餐别（早/午/晚）、人均、推荐菜 |
| **Route** | 两点间路线 | 起点、终点、距离、耗时、交通方式、路况 |
| **Budget** | 预算与费用拆分 | 总预算、四类拆分（交通/住宿/餐饮/门票）、已用、超支率 |
| **GeoLocation** | 地理位置 | 经度、纬度、地址、城市 |
| **XHSNote** | 小红书笔记（分析输入） | 标题、正文、标签、点赞、提及景点、情感倾向 |

## 4. 关键业务规则（不变量）

| 规则 | 约束 |
|------|------|
| 天数一致 | `TripPlan.days.size() == request.days`，否则触发 HITL |
| 每日有餐 | 每个 TripDay 至少含午餐与晚餐 |
| 预算红线 | `总费用 ≤ 预算 × 1.15`；超支 ≥15% 触发 HITL |
| 景点不重复 | 同一 TripPlan 内景点不重复出现 |
| 景点质量 | 推荐景点评分 ≥ 3.5/5.0，否则触发 HITL |
| 交通合理 | 景点间交通耗时 ≤ 2 小时 |
| 餐厅就近 | 餐厅距当日景点 ≤ 5km |
| 时间合理 | 每日行程在 8:00–20:00 内 |

### 4.1 业务不变量归属

| 规则类型 | 首要归属 | 说明 |
|---------|---------|------|
| 候选内容筛选（如评分阈值、情感倾向） | 对应 Agent 内部逻辑 | 由 XHS / Itinerary 等 Agent 在生成阶段优先约束 |
| 规划结果全局一致性（天数一致、无重复景点、每日有餐） | `OutputQualityGate` | 作为结果装配后的统一校验层 |
| 对外返回前的保护性校验 | Server / Facade | 兜底保证接口返回结构合法、状态一致 |
| 触发人工介入的规则（超预算、评分过低、天数不一致） | Agent + OutputQualityGate + Server 共同遵循 | Agent 负责发现，QualityGate 负责统一复核，Server 负责最终对外暴露 `requiresHumanIntervention` |

默认规则：业务不变量应先在最接近业务语义的 Agent/Service 中实现，再由 `OutputQualityGate` 做统一复核；禁止同一规则在多层无约束复制并逐渐漂移。

### 4.2 强规则 / 软规则 / HITL 规则

| 规则 | 类型 | 如何验证 |
|------|------|---------|
| `TripPlan.days.size() == request.days` | 强规则 | Agent / QualityGate / E2E 都应覆盖 |
| 每个 TripDay 至少含午餐与晚餐 | 强规则 | Itinerary 测试 + QualityGate 测试 |
| `总费用 ≤ 预算 × 1.15` | 强规则 | BudgetAgent 临界值测试 + E2E 测试 |
| 超预算 ≥15% 触发 HITL | HITL 规则 | BudgetAgent + Server 响应测试 |
| 推荐景点评分 ≥3.5 | 强规则 | XHS / Itinerary 筛选测试 |
| 景点评分 < 3.5 触发 HITL | HITL 规则 | Itinerary / QualityGate / Server 联合测试 |
| 景点间交通耗时 ≤2h | 强规则 | Route / Itinerary 测试 |
| 餐厅距景点 ≤5km | 强规则 | Itinerary 测试 |
| 每日行程在 8:00–20:00 内 | 强规则 | Itinerary 时间槽测试 |

## 5. 旅行风格与个性化（C-015 增强）

| 风格 | 偏好特征 |
|------|---------|
| 亲子（FAMILY） | 节奏慢、亲子景点、安全餐饮 |
| 情侣（COUPLE） | 浪漫景点、氛围餐厅、适度自由 |
| 独自（SOLO） | 灵活、深度、性价比 |
| 团建（TEAM） | 集体活动、大容量餐厅、统一节奏 |

### 5.1 记忆增强个性化（ADR-011）

二次规划时，系统通过 ReMe 长期记忆召回用户历史偏好与行程，注入 Agent 上下文增强推荐个性化：
- **用户画像**: travelStyle + preferences 标签累积，新规划时召回匹配偏好
- **历史行程**: confirmed 后沉淀行程摘要，同目的地/风格时召回参考
- **景点知识**: XHS 分析产出沉淀到共享知识库，丰富景点推荐依据

**关键约束**: 记忆仅作「弱增强」注入 prompt context，不覆盖 `TripPlanRequest` 硬约束（目的地、日期、预算、人数）。当历史偏好与当前请求冲突时，以当前请求为准。ReMe 不可用时降级为无记忆规划（`memoryFallback=true`），不阻塞主链路。

## 6. 业务状态机

### 6.1 规划生命周期
```
submit → analyzing → planning → assembling → budgeting → review(HITL) → confirmed
                                                            └─(用户修改)─┘
```

### 6.2 HITL 介入点
 - **计划草案确认（C-018）**：`planMode=true` 开启计划模式时，LLM 调用 `plan_write` 起草执行计划后停止，
   系统进入 `plan_review` 状态等待人工确认（CONFIRM/REJECT），确认后经 `agent.exitPlanMode(ctx)` 程序化退出 Plan Mode 并放行子 Agent 执行（T-14 官方 API）
- 超预算 ≥15%
- 行程天数 ≠ 需求天数
- 推荐景点评分 < 3.5

### 6.3 产品级自主决策边界
| Agent 可自主决定 | 必须人工确认 |
|-----------------|-------------|
| 景点搜索策略 | 最终行程方案确认 |
| 路线排序规则 | 超预算 ≥15% 时的调整方案 |
| 笔记筛选标准 | 涉及敏感/安全的内容 |
| 每日时间分配 | 最终 PDF 导出前的预览 |

> 注：此为**产品运行时**的 Agent 自主边界；**开发期** Owner Agent 的决策边界见 `.harness/agents/owner.md`。
