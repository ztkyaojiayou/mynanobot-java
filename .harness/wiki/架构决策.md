## ADR-001: Java 17 LTS

**背景**: 项目启动时 JDK 21 已发布，但生产环境主流仍是 17。

**决策**: 使用 Java 17 LTS，target=17，不依赖 preview 特性。

**理由**: 
- 17 是当前最广泛的 LTS 版本
- 已移除 `--enable-preview`（代码中未使用 Virtual Threads）
- Spring Boot 3.2 官方支持 Java 17

**后果**: 无法使用 Java 21 的 Virtual Threads、Record Patterns 等特性。

---

## ADR-002: 单模块 Maven 项目

**背景**: 原始 huazai-trip-plan 采用 Maven 多模块（9 子模块）。

**决策**: 本项目采用单模块 Maven，包内按功能分层（`com.nanobot.bus/.core/.tools/...`）。

**理由**:
- Agent 框架不涉及多团队协作
- 单模块降低构建复杂度（无模块间依赖冲突）
- 代码量 ~100 Java 文件，不需要模块隔离

**后果**: 不能按模块独立版本号发布，但不影响当前使用方式。

## ADR-003: Spring Boot 仅用于 V2

**背景**: V2 需要 HTTP/SSE/WebSocket 能力。

**决策**: Spring Boot 仅在 V2 模式使用（`@Profile("!cli")` 隔离）。V1 用内嵌 `com.sun.net.httpserver`，V3 CLI 禁用 Web 服务器。

**理由**:
- V3 CLI 对标 Claude Code，不需要 Web 容器
- V1 保留纯 Java 实现，不依赖 Spring
- `spring.profiles.active=cli` + `web-application-type=none` 彻底禁用 Web

## ADR-004: MessageBus 无外部消息队列

**背景**: 多 Agent 系统常用 Redis/RabbitMQ/Nacos 做消息通信。

**决策**: 使用 JVM 内存级 `ArrayBlockingQueue + ConcurrentHashMap`，不引入外部消息中间件。

**理由**:
- 单进程 Agent，不需要跨进程通信
- 有界队列防 OOM，ConcurrentHashMap 线程安全
- 零依赖、零配置

**后果**: 不支持分布式部署，但当前场景不需要。

---

## ADR-005: 不依赖任何 AI 框架

**背景**: Spring AI、LangChain4j 等框架提供了 Agent/Tool/LLM 调用抽象。

**决策**: 全部手写，不依赖任何 AI 框架。

**理由**:
- 学习目的：通过手写理解 Agent 底层实现
- 依赖控制：避免框架升级导致的 breaking change
- 代码量可控：~100 个 Java 文件，框架带来的便利小于维护成本

**后果**: 需要自己实现 AgentLoop、Tool 系统、Provider 适配等基础设施。

## ADR-006: ProviderFactory 策略工厂

**背景**: 需要支持 OpenAI 和 DeepSeek 两种 LLM 提供商，未来可能扩展。

**决策**: 采用 Strategy 模式，`ProviderFactory` 按模型名前缀自动匹配 Provider。每个 Provider 注册一个 `ProviderStrategy`（supports + create）。

**理由**:
- 新增 Provider 只需注册策略，不改工厂代码
- 默认兜底为 OpenAI 兼容 API
- API Key 解析优先级：配置文件 > 环境变量

**后果**: 策略注册在 `registerDefaults()` 中，可扩展但非完全配置驱动。

---

## ADR-007: AgentLoop State 模式

**背景**: AgentLoop 初始版本 ~973 行，doRestore/doCompact/doBuild 等方法集中在一个类中。

**决策**: 重构为 State 模式，每个状态独立为 `core/state/` 下的处理类，实现 `AgentState` 接口。AgentLoop 精简到 579 行。

**理由**:
- 新增状态不影响现有逻辑
- 每个 Handler 类 30-150 行，可读性强
- 扩展只需实现 AgentState + 注册

## ADR-008: 纯文件存储

**背景**: 需要会话持久化和跨会话记忆。

**决策**: 使用文件存储（JSONL + JSON + Markdown），不引入数据库。

**理由**:
- 会话数据量小（单会话几十到几百条消息）
- JSONL 增量追加写入，性能足够
- 零配置，开发者可以直接查看/编辑文件

**后果**: 不支持高并发写入，不适合生产级多用户场景。

---

## ADR-009: JLine 3 终端库

**背景**: V3 CLI 需要跨平台 Esc 键检测（Mac/Linux/Windows）。

**决策**: 引入 JLine 3 作为终端输入库。

**理由**:
- 跨平台原始按键读取（Unix stty + Windows CONIN$）
- NonBlockingReader 支持超时轮询
- 不干扰 Scanner 的正常行输入

**后果**: 增加一个外部依赖（jline-terminal 3.25.1）。

---

## ADR-010: Claude Code + Harness 协作

**背景**: 项目开发需要一个高效的 AI 协作方式。

**决策**: 全程使用 Claude Code 进行开发，引入 `.harness/` 规范体系（Spec Coding + Harness Engineering）。

**理由**:
- Claude Code 是当前最成熟的 Agent-Driven Development 工具
- `.harness/` 提供变更追踪、规则约束、技能编排的完整框架
- 本项目本身就是 Agent-Driven Development 的实践产物

**后果**: 项目结构包含 `.harness/` 目录和配套规范文件。
