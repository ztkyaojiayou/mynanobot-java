## ADR-001: 采用 AgentScope Java

**状态**: ✅ 已采纳

**背景**: 需选择 Java 多智能体框架。候选：AgentScope Java（阿里通义，企业级多 Agent）、LangChain4j（社区，原型快）、Spring AI Alibaba（Spring 生态好，多 Agent 弱）。

**决策**: AgentScope Java **`2.0.0-RC1`**，经 `agentscope-bom`（import scope）统一管理全家桶版本。

> **版本澄清（C-001 实测中央仓修订）**: 文档原记 `2.0.0+ GA`，但 Maven 中央仓核验（`io.agentscope` 全部 artifact 的 `maven-metadata.xml`）显示 **2.x 仅发布了 `2.0.0-RC1`，从无 `2.0.0` GA**；最后的 GA 稳定线是 `1.0.12`，`1.1.0` 亦仅有 RC。按护栏规则（C-001 边界：2.0.0 不可用时停下请示人类，不擅自降级/用 SNAPSHOT），经人类确认锁定 `2.0.0-RC1`。RC 为中央仓正式 release（非 SNAPSHOT），不违反「禁止 SNAPSHOT」铁律。后续 `2.0.0` GA 发布后再评估升级。

**理由**:
1. 原生 A2A 协议 + Nacos，多 Agent 协作开箱即用
2. PlanNotebook 自主任务分解，是自主决策 Agent 的关键
3. ReMe 记忆系统（短期+长期+RAG）内置
4. 内置安全沙箱，外部内容解析有隔离
5. Harness Framework 提供工作区驱动的 Agent 运行环境
6. DashScope 原生集成，LLM 链路最短

**后果**: ✅ 多 Agent 效率高、沙箱内置；⚠️ 生态年轻（2025.12 GA）、强依赖阿里云 DashScope。

**替代方案**: LangChain4j（多 Agent 需自建，否决）；Spring AI Alibaba（多 Agent 弱，否决）。

## ADR-002: Maven 多模块架构

**状态**: ✅ 已采纳

**背景**: 单模块 vs 多模块。

**决策**: Maven 多模块，按职责拆 9 个子模块（common / 5 个 agent / skills / server / tests）。

**理由**: 每个 Agent 可独立开发测试部署；依赖关系清晰避免循环（common ← skills ← agents ← server）；CI 可并行；未来可拆微服务。

**后果**: ✅ 边界清晰、并行友好；⚠️ 构建配置复杂、版本管理需纪律。

> 模块划分与依赖方向的强制约束见 `.harness/rules/工程结构.md`。

## ADR-003: A2A 协议 + Nacos 注册

**状态**: ✅ 已采纳

**背景**: 5 个 Agent 需相互协作，确定通信方式。

**决策**: AgentScope A2A 协议 + Nacos 注册中心。

**理由**: A2A 是原生协议无需适配；Nacos 一站式健康检查+服务发现+配置；Agent 松耦合；未来可平滑迁移分布式。

**后果**: ✅ 松耦合可扩缩、服务发现自动化；⚠️ 本地需启 Nacos（Docker Compose 解决）、增加网络延迟（内网可忽略）。

> 消息格式与调用约束见 `.harness/wiki/接口协议.md`。

## ADR-004: Skills 作为独立模块

**状态**: ✅ 已采纳

**背景**: 业务 Skills（景点搜索、路线规划、费用计算）是嵌入各 Agent 还是独立共享模块？

**决策**: 业务 Skills 作为独立 Maven 模块 `huazai-trip-skills`，Agent 声明式按需挂载。

**理由**: Skill 可跨 Agent 复用（如 weather-query）；独立测试不依赖特定 Agent；契合 ToolGroup 机制；新增能力=新增 Skill，不影响现有 Agent。

**后果**: ✅ 复用最大化、独立迭代；⚠️ 需设计注册发现机制、需管理版本兼容。

> ⚠️ 此处指**业务 Skills**（`huazai-trip-skills/`，Agent 运行时能力），与 `.harness/skills/`（开发技能）不同。

## ADR-006: 数据流与状态管理

**状态**: ✅ 已采纳

**背景**: 多 Agent 协作产生大量中间状态，不同生命周期数据需不同存储。

**决策**:

```yaml
短期状态: Redis（会话/中间结果/偏好，TTL 24h）
长期记忆: AgentScope ReMe 向量库（历史行程/用户画像/景点知识，永久按策略淘汰）
临时文件: 沙箱 /tmp/agentscope/{agentId}/（PDF/图片，任务结束清理）
```

**理由**: ReMe 原生支持；Redis 满足状态共享；沙箱隔离保安全；**用户数据不落盘**。

> **边界修订（ADR-012，C-017）**: 脱敏后的规划结果允许持久化到 MySQL `huazai_trip_trace` 库用于历史回溯。手机号中间 4 位 `****`、API Key `***REDACTED***`——原始 PII 不入库，不违反「不落盘」原则。审计 Trace（ADR-010）同理。

**后果**: ✅ 分层清晰、短期自动过期；⚠️ 引入 Redis 运维、ReMe 需向量库支持。

> Schema 细节见 `.harness/wiki/数据模型.md`。

## ADR-007: AI 协作方式 —— Owner Agent + Harness 流水线

**状态**: ✅ 已采纳（取代旧 ADR-007「Claude Code 作为 AI 协作工具」）

**背景**: 需确定 AI 如何深度参与开发。旧方案绑定 Claude Code 的 SDD 四阶段命令（`/spec /plan /tasks /implement /review`），实践中过重、与项目耦合度低。

**决策**: 采用 **Owner Agent + `.harness/` 六阶段开发流水线**：人类设计约束（`.harness/rules/`）→ Owner Agent 编排技能（`.harness/skills/`）→ 机器验证每一步（Checkstyle/PMD/ArchUnit/JaCoCo）→ 全程留档（`.harness/changes/`）。

**理由**:
1. 约束、技能、知识、变更全部版本化在 `.harness/`，仓库即记录系统
2. 六阶段 ①需求分析→⑥部署验证 每步有门禁，不可跳步
3. 不绑定特定 AI 工具的命令体系，方法论可移植
4. 变更状态机驱动，天然支持断点续传

**后果**: ✅ 流水线可门禁、可追溯、可恢复；⚠️ 团队需熟悉 `.harness/` 约定。

**替代方案**: SDD 四阶段 + `.claude/commands`（旧体系，已被取代，遗留文件待清理）。

> 流水线定义见 `.harness/rules/开发流程规范.md`；编排者职责见 `.harness/agents/owner.md`。

## ADR-008: 原生 vs 重写边界 —— AgentScope 原生化，Agent 入口用 HarnessAgent

**状态**: ✅ 已采纳（落实 ROADMAP 开放项 #1「C-005 原生 vs 重写边界」与 #2「AgentScope-on-SB4 冒烟」；C-006 增量 C 首次落地）

**背景**: ADR-001 决策"采用 AgentScope"，但截至 C-006 增量 B，已核实两个硬事实：①**全代码库 0 处 import `io.agentscope`**（`agentscope-core` 声明在 `huazai-trip-skills/pom.xml` 却从未使用）；②**0 处 LLM 调用**（无 DashScope/qwen），系统从未调过一次大模型。C-005 手写重造了 AgentScope 的整张能力表——`BaseAgent`/`Msg`/`AgentRegistry`/`GovernedExternalCaller`/观测/"业务 Skill"——全是 NIH。这是最差状态：**背着铁律依赖的重量与约束，却拿不到它的任何能力**。CLAUDE.md「重新实现优于包装」被误用为正当化重写核心框架（该原则本意是禁止乱引未评审的新依赖，非推倒自己定为铁律的框架）。

**决策**: 走 **AgentScope 原生**，Agent 入口统一用 **`HarnessAgent`**（`io.agentscope.harness.agent`，artifact `agentscope-harness`，经 `agentscope-bom:2.0.0-RC1` 管理，内部委托 `ReActAgent`）。**C-006 增量 C** 作为首个垂直切片先行落地：XHS 这条链由真实 `HarnessAgent` + qwen3-max + 真实 Skill（`SKILL.md`）驱动，**业务硬不变量（评分≥3.5、负面剔除、限频≤10/min、降级 fallback）继续锁在确定性 Java（`XHSNoteAnalyzer` 经 `@Tool` 暴露）**——LLM 负责解析/编排，不裁决不变量。C-005 自研栈对其余 Agent（route/itinerary/budget/supervisor）的全量原生化、Nacos 原生注册、ReMe/Redis 真实存储，拆**独立地基卡**分阶段迁移（增量 C 不动 C-005 的 A2A `Msg`/`BaseAgent` 边界）。

> **Skill 形态澄清**: AgentScope 的 "Skill" 是**提示驱动的渐进式披露**机制（`resources/skills/<name>/SKILL.md`，frontmatter `name/description/version` + 正文，经 `ClasspathSkillRepository` 加载），**不是** Java 接口；与本项目"业务 Skills"（`huazai-trip-skills/`，确定性运行时能力，见 ADR-004）分属两层。RC1 的 `SkillBox`/`MarkdownSkillParser` 等旧类已 `@Deprecated(2.0.0)`，**不使用**；走 `ClasspathSkillRepository` + `HarnessAgent.builder().skillRepository(...)` 正路。

**理由**:
1. Gate 0 spike 用硬证据排除了最大风险：`agentscope-harness:2.0.0-RC1` 解析成功、`HarnessAgent` 在 JDK21 + Spring Boot 4.0 BOM 下编译/类加载/装配 0 冲突、SB4 上下文正常启动、enforcer 无 SNAPSHOT 通过。
2. 框架对 C-005 自研的每一格都有原生对应（`ReActAgent`/`AgentBase`、原生 `Msg`、`DashScopeChatModel`、`nacos-a2a`、原生模型重试/降级），继续自研是 NIH。
3. HarnessAgent 是 Harness 项目的对口入口（workspace/memory/subagent/plan-mode），与 ADR-007 的 Harness 方法论同源；其真正甜点是 Supervisor（C-010）的 subagent/plan-mode。
4. 切片优先 + 确定性兜底：把不变量留在可 TDD 的 Java、LLM 只做其增值步骤，既真正用上框架又不牺牲可测性与回归（增量 A/B 全部回归不回退）。

**后果**: ✅ 项目首次真正运行 AgentScope + 真实 LLM；可行性已被实测证实；不变量与覆盖率门禁不被 LLM 非确定性侵蚀。⚠️ `agentscope-harness` 传递引入较重依赖（jedis/kubernetes-client/aliyun-oss 等，最小 stateless 配置不激活）；真实 LLM 链路只能经 env-gated 集成测试验证（无 `DASHSCOPE_API_KEY` 自动 skip，不伪装覆盖）；C-005 自研栈与原生的并存期需一张地基卡收口。

**替代方案**:
- 维持 C-005 全自研栈（declared-but-unused 依赖）——否决：NIH，违背 ADR-001 与铁律本意。
- 本卡内一次性推翻 C-005 全量原生化——否决：重开近 done 的地基卡、波及 C-007/8/9/10、同步→响应式契约翻转，无法在一卡内干净 TDD，风险最高。
- 仅写 ADR 定调不落代码——否决：可行性需实测（spike）而非纸面，且用户要求本卡内见到真实链路。

> XHS 切片实现见 `.harness/changes/C-006-xhs-agent/change.md` 增量 C；端口/工具/工厂见 `huazai-trip-skills` 的 `XHSAgentRunner`/`XHSAnalysisTools`/`XHSHarnessAgentFactory`。

## ADR-009: 主 LLM deepseek-v4-pro + qwen3-max 降级

**状态**: ✅ 已采纳（扩展 ADR-001 的 LLM 选型；C-006 增量 C 落地于 XHS 切片）

**背景**: ADR-001 / 技术栈铁律原定 LLM 为单一 **DashScope qwen3-max**。需求演进为：以 **deepseek-v4-pro** 为主 LLM，其不可用（超时/限频/故障）时切回 qwen3-max，提升质量与可用性。

**决策**:
- **主 LLM = deepseek-v4-pro**，经 **DeepSeek 官方 OpenAI 兼容端点**接入：`OpenAIChatModel` + `DeepSeekFormatter` + `baseUrl=https://api.deepseek.com` + 环境变量 `DEEPSEEK_API_KEY`。
- **降级 LLM = qwen3-max**，经 DashScope（`DashScopeChatModel` + `DASHSCOPE_API_KEY`）。
- **切换机制走 AgentScope 原生** `HarnessAgent.builder().model(deepseek).fallbackModel(qwen)`——运行期主模型失败自动切回，不自建重试编排。
- **密钥策略**：`DEEPSEEK_API_KEY`（主）与 `DASHSCOPE_API_KEY`（降级）均纳入启动期 `EnvironmentValidator.REQUIRED_KEYS` fail-fast；密钥禁硬编码、走环境变量/配置中心。工厂 `XHSHarnessAgentFactory.fromEnvironment` 容错：缺 deepseek key 时降级为 qwen 单模型（便于 CI/本地）。
- **模型 ID 可配置**：`deepseek-v4-pro`/`qwen3-max` 作为默认常量（`XHSHarnessAgentFactory.DEFAULT_PRIMARY_MODEL`/`DEFAULT_FALLBACK_MODEL`），接入路一致时改 ID 即可，不动代码结构。

**理由**: 主/降级双模型提升可用性与质量；用 HarnessAgent 原生 `fallbackModel` 而非自建，符合 ADR-008 原生化方向；DeepSeek OpenAI 兼容端点 + `DeepSeekFormatter` 是 AgentScope 一等支持的接入路。

**后果**: ✅ 主模型故障可自动降级、不中断主链路；模型可配置易替换。⚠️ 新增 `DEEPSEEK_API_KEY` 与第二个 provider/端点（两套密钥）；真实主链路验证需 `DEEPSEEK_API_KEY`（env-gated `XHSHarnessAgentFactoryIT`，无 key 自动 skip）；`deepseek-v4-pro` 具体模型 ID 以接入环境为准（默认填入，可改配置）。

**替代方案**:
- 经 DashScope 托管的 deepseek（复用单密钥）——否决（本次）：用户选择 DeepSeek 官方端点以拿到一等模型能力。
- 自建主/降级 try-catch 切换——否决：HarnessAgent 原生 `fallbackModel` 已覆盖，自建即 NIH（违 ADR-008）。

> 落地见 `XHSHarnessAgentFactory`（双模型组装）、`EnvironmentValidator`（密钥 fail-fast）、`.env.example`；铁律表（CLAUDE.md/AGENTS.md/README.md/owner.md）与 `接口协议.md` §4 已同步。

## ADR-010: 审计追踪采用 MySQL + 异步写入 + 脱敏

**状态**: ✅ 已采纳（C-010，2026-06-16）

**背景**: SupervisorAgent 编排全链路需审计回溯能力（谁委派了谁、LLM 调用详情、编排汇总），但 OTel/Metrics 侧重实时指标而非结构化审计查询。需一个不影响主链路性能的审计持久化方案。

**决策**:
- **数据库**: MySQL 8.0，独立库 `huazai_trip_trace`，三表通过 `traceId` 关联：
  - `trip_agent_delegation_log` — 子 Agent 委派记录
  - `trip_llm_call_log` — LLM 调用记录
  - `trip_orchestration_summary` — 编排汇总记录
- **接口隔离**: `TracePersistenceService` 接口放 `huazai-trip-skills`，`MySqlTracePersistenceService` 实现放 `huazai-trip-agent-supervisor`。`SupervisorAgent` / `TripOrchestrationService` 仅依赖接口。
- **异步写入**: 有界线程池（core=2, max=4, queue=1000, CallerRunsPolicy），写入失败仅 WARN 日志 + metrics 计数，**不阻塞、不抛异常到主链路**。
- **脱敏**: `SensitiveDataMasker` 正则替换 API Key（DEEPSEEK_API_KEY / DASHSCOPE_API_KEY）为 `***REDACTED***` 后落库。
- **连接池**: HikariCP，经 `TraceDataSourceConfig`（`@Configuration` + `@ConditionalOnProperty`）从 `trip.trace.datasource.*` 配置项装配。
- **测试**: H2 内存库单测覆盖核心逻辑；JaCoCo 排除 `TraceThreadPoolConfig`（纯配置类）。

**理由**: MySQL 提供结构化查询（按 traceId/planId/agentId/时间范围检索）；异步写入不影响编排主链路延迟（P95 < 30s）；接口隔离确保 `huazai-trip-agent-supervisor` 不反向依赖 server；脱敏防止密钥泄漏到审计日志。

**后果**: ✅ 全链路审计可追溯、主链路性能无影响。⚠️ 新增 MySQL 基础设施依赖（本地开发需 Docker MySQL 或设置 `TRACE_DB_*` 环境变量；CI test profile 通过 `trip.trace.datasource.enabled=false` 关闭）；三表数据量随编排次数线性增长（建议保留策略 90 天）。

**替代方案**:
- 纯文件日志（Logback/ELK）——否决：非结构化，难以按 traceId 检索。
- Redis Streams ——否决：内存成本高，持久化弱。
- 直接嵌入 OTel span attributes ——否决：span 有大小限制，不适合大段文本摘要。

> 落地见 `TracePersistenceService`（接口）、`MySqlTracePersistenceService`（JDBC 实现）、`TraceDataSourceConfig`（Spring 装配）、`SensitiveDataMasker`（脱敏）、`sql/schema.sql`（DDL）、`application*.yml`（配置）。

## ADR-011: 记忆层选型 —— ReMe Server + agentscope-extensions-reme

**状态**: ✅ 已采纳（C-015，2026-06-30）

**背景**: ADR-006 三层存储分层预留了「长期记忆」层位（ReMe 向量库），但未落地实现。C-015 需要为二次规划个性化提供用户偏好/历史行程/景点知识的召回能力。需选型长期记忆存储方案。

**决策**:
- **存储**: 采用 **ReMe Server**（Python 服务，`reme-ai[core]`），通过 Docker 独立部署（`docker/reme/Dockerfile`），HTTP API 提供记忆读写。
- **Java 客户端**: 使用 AgentScope 官方扩展 **`agentscope-extensions-reme`**（`io.agentscope:agentscope-extensions-reme`，经 `agentscope-bom:2.0.0-RC4` 管理），核心类 `ReMeClient`。
- **隔离模型**: 以 **workspace = userId** 实现用户维度的记忆隔离；共享景点知识使用固定 workspace `attraction_kb`。
- **三类记忆**: ① 行程摘要（confirmed 后异步写入用户 workspace）；② 用户画像（travelStyle + preferences 累积更新）；③ 景点知识（XHS 分析产出沉淀到 `attraction_kb`）。
- **召回策略**: 新规划时通过 `ReMeClient.search()` 按相似度召回，封装为 `MemoryContext` 注入 Agent 上下文。记忆仅作弱增强，不覆盖 `TripPlanRequest` 硬约束。
- **降级**: ReMe 不可用时降级为「无记忆」常规规划（`memoryFallback=true`），不阻塞主链路。
- **Embedding**: ReMe Server 内置，配置 `EMBEDDING_API_KEY`（DashScope `text-embedding-v3`），Java 侧不直接调用 Embedding。
- **数据清除**: `MemoryClearService` 通过 HTTP DELETE 清除用户 workspace，支撑合规。

**理由**:
1. ReMe 是 AgentScope 原生记忆方案（ADR-006 预留位），官方扩展一等支持，符合 ADR-008 原生化方向
2. 不自建向量存储/Embedding 调用链路，复用 ReMe Server 内置能力（「无聊技术」原则）
3. workspace 隔离天然支持多用户，不需额外租户逻辑
4. 降级设计确保 ReMe 故障不影响主链路可用性

**后果**: ✅ 二次规划可召回历史偏好，个性化增强；官方扩展维护成本低。⚠️ 新增 Python 服务依赖（Docker 部署）；ReMe Server 的 Embedding 配置需 `EMBEDDING_API_KEY`（DashScope）；真实召回效果依赖 ReMe 的相似度算法。

**替代方案**:
- 自建 Qdrant/Chroma 向量库 + 自管 Embedding——否决：引入未评审基础设施，与「无聊技术」原则和 ADR-006 预留位冲突。
- Redis 存储长期记忆（扩展现有 Redis）——否决：Redis 不支持向量相似度检索，无法满足按偏好/风格召回。
- 仅用 Redis 缓存 + 精确匹配——否决：无法实现语义相似度召回，个性化效果差。

> 落地见 `MemoryRecallService`/`MemoryWriteService`/`MemoryClearService`（skills/memory）、`MemoryContext`（common/dto）、`ReMeConfig`（server/config）、`docker/reme/Dockerfile`。

## ADR-012: 历史规划持久化 —— MySQL 复用 + 异步写入 + 脱敏 + planId 序列号

**状态**: ✅ 已采纳（C-017，2026-07-04）

**背景**: ADR-006 规定「用户数据不落盘」，ADR-010 引入 MySQL `huazai_trip_trace` 库用于审计追踪。C-017 需要持久化历史规划结果（行程/景点/餐饮/路线/费用），支持用户回溯对比历史方案，同时需修复 ADR-010 中 `TracePersistenceService` 注入断裂导致审计三表从未实际写入的问题。

**决策**:
- **数据库**: 复用 ADR-010 的 MySQL `huazai_trip_trace` 库及 `TraceDataSourceConfig` HikariCP 连接池，不新增独立数据源。
- **业务表**: 6 张表（`trip_plan_record`/`trip_day_record`/`trip_attraction_record`/`trip_meal_record`/`trip_route_record`/`trip_budget_record`），通过 `plan_id` 关联。
- **接口隔离**: `TripPlanPersistenceService` 接口放 `huazai-trip-skills`（与 `TracePersistenceService` 平级），`MySqlTripPlanPersistenceService` 实现放 `huazai-trip-agent-supervisor`。
- **异步写入**: 复用 `TraceThreadPoolConfig.createTraceExecutor()`（core=2, max=4, queue=1000, CallerRunsPolicy），写入失败仅 WARN + metrics，不阻塞主链路。
- **挂载点**: `TripPlanFacade.executeOrchestration()` 末尾异步调用，捕获 CONFIRMED 和 FAILED 两种终态。
- **脱敏**: 复用 `SensitiveDataMasker`（API Key 脱敏），新增手机号 Pattern 委托 `MaskUtils.maskPhone()`。
- **历史查询**: Redis-first（`PlanCacheService` ZSet + PlanSummary，30 天 TTL）→ MySQL-fallback。
- **planId 序列号**: 从纯内存 `AtomicInteger` 改为「Redis 优先 + 内存兜底」，key `trip:plan:seq:{yyyyMMdd}` TTL 48h。
- **ADR-006 修订**: 脱敏后的规划结果允许持久化到 MySQL 用于历史回溯，不违反「PII 不落盘」原则——手机号中间 4 位 `****`，API Key `***REDACTED***`，原始 PII 不入库。

**理由**:
1. 复用 ADR-010 已有 MySQL 基础设施（连接池、线程池、数据源配置），不增加运维复杂度
2. 原生 JDBC + PreparedStatement 与 `MySqlTracePersistenceService` 保持一致，不引入 JPA/MyBatis
3. 异步写入 + 降级跳过确保持久化故障不影响主链路
4. 脱敏后落库满足合规要求（PII 不以原文入库）

**后果**: ✅ 用户可回溯对比历史规划；审计 Trace 注入修复后首次真正落库；planId 跨重启不重置。⚠️ 6 张业务表数据量随规划次数线性增长（建议保留策略 180 天）；软删除仅逻辑标记，需定期归档。

**替代方案**:
- 引入 JPA/MyBatis——否决：增加依赖，与现有 JDBC 风格不一致，违反「无聊技术」原则。
- 新增独立 MySQL 库——否决：增加运维复杂度，且 `huazai_trip_trace` 库已就位。
- 仅用 Redis 存储历史——否决：Redis TTL 30 天后数据丢失，无法长期回溯。

> 落地见 `TripPlanPersistenceService`（接口）、`MySqlTripPlanPersistenceService`（JDBC 实现）、`TripPlanHistoryService`（查询）、`PlanIdSequenceService`（序列号）、`SensitiveDataMasker`（手机号脱敏扩展）。
