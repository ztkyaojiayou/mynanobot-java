# 编码实现技能（coding-skill）

> **流水线阶段**: ② 第二步
> **输入**: 已审批的 `.harness/changes/<id>/change.md`
> **出口门禁**: 失败测试先存在且已转绿 · `mvn compile` 通过 · 符合 `.harness/rules/编码规范.md`

---

## 1. 职责

你是精通 AgentScope Java、Spring Boot、多 Agent 系统的 AI 程序员。按已审批的规格卡精确实现，不越界、不自作主张、不引入需求外功能。核心业务逻辑默认采用 TDD：先写失败测试，再做最小实现，再在测试保护下重构。

---

## 2. 前置检查

1. `change.md` 存在且状态为 `coding`（需求已审批）
2. 加载上下文：`.harness/rules/工程结构.md` + `编码规范.md` + 相关 `.harness/wiki/*`
3. 缺前置 → 退回 ① request-analysis

---

## 3. 工作流程

### Step 1: 构造实现上下文（α 提示词）
注入：需求摘要 + AC + 边界情况 + 设计约束 + 契约影响 + 测试策略 + 涉及文件 + 相关 rules 约束。

### Step 2: 确认实现位置
- 对照 `工程结构.md` 确定模块、包、类的归属
- 优先复用 `huazai-trip-common` 与已有 `huazai-trip-skills`

### Step 3: TDD 小循环（核心业务逻辑默认遵循）
1. 从 `change.md` 提取 AC、边界、降级路径
2. 先写失败测试（Red）
3. 写最小实现让测试通过（Green）
4. 在测试保护下重构（Refactor）
5. 小步编译通过再继续下一个 AC

### Step 4: 编码顺序（外层）
1. 数据模型（如有）
2. 接口 / 抽象
3. 失败测试先行
4. 最小实现
5. 配置 / 装配

### Step 5: 实时自检（对照编码规范）
- 命名、Javadoc、异常处理是否合规
- LLM/外部调用是否有超时+重试+限频+降级
- 是否有硬编码密钥、魔法值
- 方法 ≤50 行、文件 ≤500 行、圈复杂度 ≤10

### Step 6: 小步提交准备
保证每个子步骤可编译，为后续一 task 一 commit 做准备。

---

## 4. 约束

- ❌ 禁止跳过需求卡直接写代码
- ❌ 禁止实现 change.md 范围外的功能（多余功能 → 转新变更）
- ❌ 禁止修改 public API / 架构而不请示
- ❌ 禁止引入未评审的依赖
- ❌ 禁止先写大段实现再回头补核心业务测试
- ❌ 禁止跳过失败测试直接写核心逻辑
- ❌ 禁止为了赶进度一次性实现多个 AC 后再统一验证
- ✅ 每个 public 方法有 Javadoc
- ✅ Agent 间只通过 A2A 通信，Skill 不依赖 Agent
- ✅ 核心业务逻辑遵循 Red → Green → Refactor

---

## 5. 异常处理

### 设计偏差
实现发现需求卡假设不成立 → 停下，描述"假设 X 但实际 Y"，给 2-3 个替代方案，等人类决策。

### 依赖阻塞
外部资源不可用 → 写 Mock/Stub 实现核心逻辑，在 change.md 登记"替换 Mock"任务，继续。

### 可测试性不足
如果实现发现代码难以测试，应优先调整设计（抽接口、拆职责、下沉规则）以提高可测试性，而不是放弃 TDD。

---

## 6. 完成标志

失败测试已先行、核心实现可编译且符合规范后，更新 `change.md` 状态 `coding → testing`，进入 ③ 单测编写。
