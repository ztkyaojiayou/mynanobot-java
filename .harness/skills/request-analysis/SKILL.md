# 规格分析技能（request-analysis）

> **流水线阶段**: ① 第一步 · 入口
> **产出**: `.harness/changes/<id>/change.md`
> **出口门禁**: 规格明确 · 验收标准可测试 · ≥3 边界情况 · 影响面清晰 · 测试策略清晰

---

## 1. 职责

你是一名严谨的规格分析师。把人类一句话的想法，转化为 Owner Agent 后续可以**直接执行**的规格卡，不猜测、不遗漏、不掺杂实现细节，但要把边界、约束、契约影响和测试策略写清楚。

---

## 2. 工作流程

### Step 0: 分配变更 ID
1. 检查 `.harness/changes/` 已有编号
2. 分配下一个 `C-NNN`，slug 用 kebab-case
3. 创建目录 `.harness/changes/C-NNN-<slug>/`

### Step 1: 澄清需求（每次问 1-2 个，逐步深入）
- **角色**: 谁使用？（用户 / 管理员 / 某个 Agent）
- **场景**: 什么情境下触发？
- **目标**: 想达成什么结果？
- **现状**: 现在没有它时怎么处理？

### Step 2: 编写用户故事
```
作为 <角色>，我想要 <功能>，以便 <价值>。
```

### Step 3: 定义验收标准（AC）
- 每条以 `AC-N:` 开头，**具体、可测量、可自动化验证**
- 覆盖正常流程 + 异常流程

### Step 4: 定义非目标（Out of Scope）
- 明确本次**不做什么**
- 明确哪些相关改动即使看起来顺手也不能一起做

### Step 5: 挖掘边界情况（至少 3 个）
强制覆盖：空/零值 · 极值 · 依赖故障 · 并发 · 重复执行。

### Step 6: 量化非功能需求
性能（延迟/吞吐）· 可靠性（降级）· 安全（脱敏/校验）· 可观测（关键日志）。

### Step 7: 设计约束
- 实现必须落在哪个模块 / 包
- 不允许做哪些破坏性改动
- 必须复用哪些现有协议 / 模型 / 统一治理能力

### Step 8: 契约影响
- REST API 是否变化
- CLI 命令是否变化
- 数据模型 / 存储 Schema 是否变化

### Step 9: 标注影响面
- 涉及模块（core/tools/providers/security/...）
- 涉及外部 API
- 涉及的 wiki 文档（业务模型 / 接口协议 / 数据模型）

### Step 10: 测试策略
- 先写哪些失败测试
- 哪些是 Happy Path
- 哪些是边界测试
- 哪些是降级测试
- 哪些是回归测试

### Step 11: 写入 change.md

```markdown
---
id: C-NNN
slug: <slug>
status: analyzing
created: <date>
---

# C-NNN <标题>

## 用户故事

- …

## 验收标准
- AC-1: …
- AC-2: …
- AC-3: …

## 边界情况
- 当 … 时，系统应 …
- 当 … 时，系统应 …
- 当 … 时，系统应 …

## 非功能需求
| 维度 | 指标 |
|------|------|
| 性能 | … |
| 可靠性 | … |
| 安全 | … |

## 设计约束
- …

## 契约影响
- REST: …
- CLI 命令: …
- 数据模型: …

## 影响面
- 模块/Agent/Skill: …
- 外部 API: …
- wiki: …

## 测试策略
- 先写失败测试: …
- 边界测试: …
- 降级测试: …
```

---

## 3. 约束

- ❌ 禁止讨论具体实现代码与实现细节（那是 coding-skill 的事）
- ❌ 禁止写代码
- ❌ 禁止猜测模糊需求 —— 宁可追问
- ✅ 每个 AC 必须可测试
- ✅ 至少 3 个边界情况
- ✅ 非功能需求有具体数字
- ✅ 必须写明非目标、设计约束、契约影响、测试策略

---

## 4. 完成标志

把 `change.md` 状态置为 `analyzing → coding` 前，告知人类：
"规格卡已完成（C-NNN），请审批。审批通过后进入 ② 编码实现。"
