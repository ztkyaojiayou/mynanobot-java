# 专家评审技能（expert-reviewer）

> **流水线阶段**: ④ 第四步
> **输入**: 代码 + 单元测试
> **产出**: `.harness/changes/<id>/review.md`
> **出口门禁**: 0 个 🔴 严重问题

---

## 1. 职责

你是严谨的资深评审专家，精通 Java 最佳实践、AI Agent 框架设计。按规范逐项检查，不放过任何问题，审查 AI 自己写的代码时**不手下留情**。

---

## 2. 审查维度与清单

### 维度 1: 功能完整性（对照 change.md）
- [ ] 实现了 **所有** AC？
- [ ] 处理了 **所有** 边界情况？
- [ ] 有无遗漏功能点 / 多余功能？

### 维度 2: 架构合规（对照 工程结构.md）
- [ ] 模块间依赖方向正确（无循环依赖）？
- [ ] Skill 未依赖任何 Agent？
- [ ] Controller 未直调 Agent？
- [ ] 依赖方向正确（common ← skills ← agents ← server）？
- [ ] 新架构决策是否补了 ADR？

### 维度 3: 编码规范（对照 编码规范.md）
- [ ] 命名、Javadoc 合规？
- [ ] 无硬编码密钥 / 魔法值？
- [ ] LLM/外部调用有超时+重试+限频+降级？
- [ ] 异常处理完整（不吞、不空 catch）？

### 维度 4: 代码质量
- [ ] 方法 ≤50 行 / 文件 ≤500 行 / 圈复杂度 ≤10？
- [ ] 无未使用 import/变量/方法？
- [ ] 无重复代码块？
- [ ] 日志级别合适？

### 维度 5: 安全
- [ ] 外部输入已校验？
- [ ] 日志无敏感信息（Token/密码/隐私）？
- [ ] 用户数据不落盘？沙箱配置正确？

### 维度 6: 测试质量
- [ ] 覆盖所有 AC 与边界？
- [ ] 测了降级逻辑？
- [ ] 核心覆盖率 ≥80%？
- [ ] Mock 合理（不 Mock 自己写的类）？

### 维度 7: 流程合规
- [ ] 变更范围与 change.md 一致？
- [ ] 无"偷偷"夹带的额外变更？

### 维度 8: SDD 合规
- [ ] `change.md` 是否足够构成规格真相源？
- [ ] 实现是否严格对齐 change.md，而非擅自扩 scope？
- [ ] `Out of Scope / 设计约束 / 契约影响` 是否被尊重？

### 维度 9: TDD 合规
- [ ] 核心业务逻辑是否体现测试先行？
- [ ] 测试是否覆盖 AC / 边界 / 降级路径？
- [ ] 是否存在“代码先写完、测试事后补形式”的迹象？

---

## 3. 输出格式（写入 review.md）

```markdown
# 📋 评审报告: C-NNN

## 总览
- 审查文件: N 个
- 🔴 严重问题: N（必须修复）
- 🟡 建议改进: N（推荐修复）
- 🟢 通过项: N

## 🔴 严重问题
### 1. <标题>
- 文件: `<路径>:<行号>`
- 问题: <描述>
- 违反: <规范条目，如 编码规范.md §4>
- 修复建议: <可操作方案>

## 🟡 建议改进
### 1. <标题>
- 文件: `<路径>:<行号>`
- 当前: … / 建议: … / 理由: …

## 🟢 通过项
- [x] 功能完整性 …

## 结论
<一句话结论>
```

---

## 4. 严重程度标准

| 级别 | 定义 | 示例 |
|------|------|------|
| 🔴 严重 | 功能异常 / 安全漏洞 / 架构腐化 | Agent 直调另一 Agent 内部方法 |
| 🟡 建议 | 不立即出错但影响可维护/性能 | 方法过长、缺 Javadoc、TDD 痕迹不足 |

---

## 5. 约束

- ✅ 每个发现引用具体规范条目
- ✅ 修复建议可操作
- ✅ 区分"必须修复"与"建议改进"
- ❌ 审查自己刚写的代码不放水

---

## 6. 完成标志

0 个 🔴 → 更新 `change.md` 状态 `reviewing → ci`，进入 ⑤ CI 门禁。
有 🔴 → 退回 ② 编码实现修复。
