# 编码规范

> **作用域**: mynanobot-java 全模块 Java 代码标准与约定。
> **基线**: Alibaba Java Coding Guidelines（阿里巴巴 Java 开发手册）。
> **状态**: 强制 · 由 Checkstyle + PMD 机械化检查

---

## 1. 命名约定

| 元素 | 规则 | 示例 |
|------|------|------|
| 类 / 接口 | 大驼峰 UpperCamelCase | `RoutePlanningService` |
| 方法 / 变量 | 小驼峰 lowerCamelCase | `calculateTotalCost` |
| 常量 | 全大写 + 下划线 | `MAX_CALL_DEPTH` |
| 包 | 全小写、单数 | `com.nanobot.route.tool` |
| Agent 类 | 以 `Agent` 结尾 | `BudgetAgent` |
| Tool 类 | 以 `Tool` 结尾 | `BaiduMapTool` |
| Service 类 | 以 `Service` 结尾 | `CostModelService` |
| 测试类 | 被测类名 + `Test` | `BudgetAgentTest` |
| 布尔变量 | 禁止 `is` 前缀（POJO 字段） | `boolean deleted`（非 `isDeleted`） |

- 禁止拼音与英文混用、禁止无意义缩写（`a1`、`tmp2`）。
- 禁止魔法值：所有字面量常量化（放 `constant/` 或枚举）。

## 2. 注释与文档

- **所有 public 类与方法必须有 Javadoc**，说明职责、参数、返回值、异常。
- Javadoc 用第三人称陈述（"Returns the…"风格的中文等价："计算…并返回…"）。
- 复杂逻辑用行内注释解释**为什么**，而非**做什么**。
- `TODO` / `FIXME` 必须转化为 `.harness/changes/` 中的任务条目，禁止长期遗留。

```java
/**
 * 估算给定行程的总费用，按交通/住宿/餐饮/门票四类拆分。
 *
 * @param plan 待估算的行程方案，不可为 null
 * @return 费用明细；当数据缺失时返回保守上界估算
 * @throws IllegalArgumentException 当 plan 为 null
 */
public CostBreakdown estimate(TripPlan plan) { ... }
```

### 2.1 Lombok 使用策略
- ✅ 允许: `@Builder`、`@Slf4j`、`@RequiredArgsConstructor`
- ❌ 禁止: `@Data`（显式编写 getter/setter/toString，避免隐式 equals/hashCode 陷阱）

### 2.2 日志级别约定
| 级别 | 用途 |
|------|------|
| DEBUG | 开发调试细节 |
| INFO | 关键流程节点（Agent 输入/输出、任务分解、HITL 结果） |
| WARN | 降级、重试、限频命中 |
| ERROR | 需人工介入的失败 |

结构化 JSON 日志，含 `timestamp / level / agent / traceId / message / context`。

### 2.3 最小日志字段规范
所有关键日志（Agent 输入/输出、A2A 通信、外部调用、HITL、降级）至少包含以下字段中的适用子集：
- `traceId`
- `planId`
- `agentId`
- `taskType`
- `msgId`
- `changeId`（开发期日志/脚本/验证任务）
- `fallback`

若字段在当前上下文不存在，可省略；若存在却未记录，视为可观测性缺陷。

## 3. 异常处理

| 规则 | 说明 |
|------|------|
| 不吞异常 | 禁止空 catch；至少记录日志或转译 |
| 不滥用 RuntimeException | 业务异常继承项目 `BaseException` 体系 |
| 异常分层 | `AgentCallException` / `RateLimitException` / `LLMTimeoutException` 等语义化 |
| 资源释放 | 用 try-with-resources，禁止裸 `finally` 手动 close |
| 异常信息 | 包含上下文（哪个 Agent、哪个 task、关键参数），但**禁止泄露密钥/隐私** |

```java
// ❌ 禁止
try { call(); } catch (Exception e) {}

// ✅ 正确
try {
    return llmClient.call(prompt);
} catch (TimeoutException e) {
    log.warn("LLM 调用超时, agent={}, fallback to default", agentId, e);
    return defaultTemplate();
}
```

## 4. LLM / 外部调用约束（强制）

每一次 LLM 或外部 API 调用必须具备：

| 维度 | 要求 |
|------|------|
| 超时 | 30s |
| 重试 | 3 次，指数退避（1s / 2s / 4s） |
| 限频 | 小红书 ≤10/min · 地图 ≤30/min · 通义 ≤60/min（Guava RateLimiter） |
| 降级 | 必须有默认值/缓存/离线估算，禁止裸抛到用户 |
| Token 上限 | 输入 ≤8000 + 输出 ≤4000 |

### 4.1 统一外部调用治理
- 外部调用能力必须尽量收敛到统一封装层（如 `BaseTool`、统一 Client Wrapper、Interceptor、Policy 组件），禁止每个 Agent/Service 各自散写一套超时/重试/限频/降级。
- 统一封装层至少负责：超时、重试、限频、fallback、错误分类、最小日志字段。
- 同一外部依赖（如小红书、地图、LLM）在项目内应只有一套主治理策略，局部特例必须在 change 或 ADR 中说明。
- 若出现多个 Agent 分别复制同样的重试/限频逻辑，视为规范违背与技术债。

## 5. 空安全与防御式编程

- public 方法入参做非空/范围校验（用 `Objects.requireNonNull` 或 `ValidationUtils`）。
- 返回集合用空集合而非 `null`。
- `Optional` 用于可能缺失的返回值，禁止 `Optional` 作为字段或入参。
- 避免 NPE：链式取值前判空，或用 `Optional.map`。

---

## 6. 并发与状态

- Agent 调用链深度 ≤ 5（防死锁，`AgentCallGuard` 守护）。
- 共享状态走 Redis / ReMe，禁止可变静态字段持有业务状态。
- `ThreadLocal` 用完必须 `remove()`。
- 不可变消息（`Msg`）传递，禁止跨 Agent 共享可变对象。

---

## 7. 安全红线

| 红线 | 规则 |
|------|------|
| 密钥 | API Key/Token/密码 → 环境变量 或 配置中心，**禁止硬编码** |
| 日志 | 禁止打印 Token、密码、用户隐私（手机号/身份证脱敏） |
| 用户数据 | 不落盘，仅 Redis + 内存 |
| 输入校验 | 所有外部输入（小红书内容、用户请求）必须校验后使用 |
| 沙箱 | Agent 文件操作限定 `/tmp/agentscope/{agentId}/` |

### 7.1 绝对禁止事项
- ❌ 在核心逻辑中用反射绕过类型安全
- ❌ 硬编码 API Key / Token / 密码
- ❌ 在 Agent 中直接操作数据库（必须走 Repository 接口）
- ❌ 绕过 A2A 协议直接调用其他 Agent 的内部方法
- ❌ 在非测试代码中用 `Thread.sleep()` 作为等待策略
- ❌ 引入非 Apache 2.0 / MIT 兼容许可证的依赖

## 8. 代码质量量化指标

| 指标 | 阈值 | 守护 |
|------|------|------|
| 文件长度 | ≤ 500 行 | Checkstyle |
| 方法长度 | ≤ 50 行 | Checkstyle |
| 圈复杂度 | ≤ 10 | Checkstyle |
| 核心逻辑覆盖率 | ≥ 80% | JaCoCo |
| 重复代码块 | 禁止（提取复用） | PMD CPD |
| 未使用 import/变量 | 禁止 | Checkstyle / PMD |

---

## 8.1 测试优先编码约定

- Agent 核心逻辑、业务规则判断、降级策略、OutputQualityGate 默认采用测试先行。
- 每条 AC 应有对应测试映射；每条关键降级路径应有专门测试。
- 测试优先验证**行为与结果**，而不是实现细节。
- 不以“最终有测试”替代“测试先行”；核心逻辑若只有事后补测，视为过程不合规。

## 8.2 可测试性设计约定

- 外部依赖通过接口 / Tool / Client Wrapper 隔离，避免散落在核心业务逻辑中。
- 避免静态耦合、隐藏时间依赖、全局可变状态，使核心规则可独立测试。
- 若某段核心逻辑难以编写测试，应优先调整设计结构，而不是降低测试要求。

## 9. 提交规范（Conventional Commits）

```
<type>(<scope>): <CHANGE-ID> <简短描述>

type:  feat | fix | refactor | test | docs | chore
scope: common | skills | server | <agent-name> | 001-xhs-analysis …

示例:
  feat(004-budget): C-012 实现超支预警 ≥15% 触发 HITL
  test(002-route-planning): C-013 补充离线估算降级测试
  fix(common): C-014 修复 GeoUtils 距离计算精度问题
```

**粒度**: 一个任务一次 commit，message 必须含变更 ID（关联 `.harness/changes/`）。

---

## 10. 简单优先（YAGNI）

- 能用简单方案就不用复杂方案；优先复用 `huazai-trip-common`。
- 不为"未来可能的需求"写代码；每个抽象必须有"现在"的价值。
- 不引入大型第三方库只为一个小功能；重新实现优于包装第三方。
