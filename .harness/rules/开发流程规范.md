# 开发流程规范

> **作用域**: 定义 mynanobot-java 从需求到上线的开发流水线、技能调度顺序、阶段门禁、变更追踪。
> **执行者**: `.harness/agents/owner.md`（Owner Agent）
> **状态**: ①~④ 阶段强制执行，⑤⑥ 阶段为未来目标

---

## 1. 流水线总览

```
                                 ┌──── 每一步对照 .harness/rules/ ────┐
                                 │                                    │
 人类意图                        ▼                                    │
    │   ① request-analysis ──▶ ② coding-skill ──▶ ③ unit-test ──▶ ④ review ──▶ ⑤ CI(未来) ──▶ ⑥ deploy(未来) ──▶ 交付
    │   规格构建                测试先行实现        测试完善           专家评审
    ▼
 .harness/changes/<id>/   （全程留档，状态机驱动）
```

每个阶段对应 `.harness/skills/` 下一个技能，**严格按序，不可跳步**。

---

## 2. 阶段定义与门禁

| 阶段 | 技能 | 输入 | 产出 | 出口门禁 |
|------|------|------|------|---------|
| ① 需求分析 | `request-analysis` | 一句话意图 | 规格卡 + AC + 边界情况 + NFR | 规格明确、AC 可测试、≥3 边界情况 |
| ② 编码实现 | `coding-skill` | 已确认的规格卡 | 失败测试 + 最小实现 | 测试通过 + `mvn compile` 通过 |
| ③ 单测编写 | `unit-test-write` | 实现代码 | 边界/降级/回归测试 | 核心覆盖率 ≥80% |
| ④ 专家评审 | `expert-reviewer` | 代码 + 测试 | 评审报告 | 0 个 🔴 严重问题 |
| ⑤ CI 门禁 | `unit-test-ci` | 完整变更 | CI 报告 | 59/59 全绿（当前手动 `mvn test` 等效） |
| ⑥ 部署验证 | `deploy-verify` | 通过 CI 的构建 | 验证报告 | 冒烟 + 健康检查通过 |

> **门禁原则**: 任一 Gate 不通过 → 退回修复或请示人类，**禁止进入下一阶段**。

---

## 3. 变更追踪

每个需求 = 一个变更，分配唯一 ID `C-NNN`：

```
.harness/changes/C-NNN-<slug>/
├── change.md          # 变更卡（用户故事 + AC + 边界 + NFR）
├── review.md          # 设计评审记录（④ 产出）
└── verify.md          # 验证记录（mvn test 结果）
```

**变更状态机**:

```
draft → analyzing → coding → testing → reviewing → ci → verifying → done
                                                  └──(打回)──┘
```

## 4. 人机协同协议

### 4.1 人类职责
- 定义需求、做架构决策、确定优先级、最终验收
- 审批 ① 需求卡、④ 评审报告中的关键决策

### 4.2 Owner Agent 职责
- 编排技能、编写代码与测试、更新文档、自审计
- 不做架构决策（只给建议）、不改变需求范围

### 4.3 必须暂停请示的触发点
1. 需求模糊到无法确定实现方式
2. 设计方案在实现时被证伪
3. 需要新增依赖 / 改 API / 改架构
4. 发现安全风险
5. 触碰生产配置或密钥

---

## 5. 特殊场景裁剪

| 场景 | 裁剪规则 |
|------|---------|
| **Hotfix** | 可简化 ①，但必须最小变更卡；修复后补全 |
| **Spike（技术探索）** | 可跳过完整 ①②③，结论沉淀为 ADR |
| **棕地修改（改已有功能）** | 先更新对应 change/verify，再走 ②~④ |
| **纯文档变更** | 仅走 ①（变更卡）+ ④（评审），无需测试 |

---

## 6. 提交规范

```
<type>: <简短描述>

type: feat | fix | refactor | test | docs | chore

示例:
  feat: Plan Mode /plan approve 审批执行
  refactor: AgentLoop 状态机抽取为 State 模式
  docs: README 补充部署和使用指南
```

**粒度**: 一个功能/bug 一次 commit，`mvn test` 通过后提交。

---

## 7. 质量检查清单（每个变更收口）

- [ ] `mvn compile` 通过
- [ ] `mvn test` 59/59 全绿
- [ ] `.harness/changes/<id>/change.md` 状态 = `done`
- [ ] 涉及的文档已同步（README / NANOBOT.md / 架构文档）
- [ ] 无残留 TODO/FIXME 标记
