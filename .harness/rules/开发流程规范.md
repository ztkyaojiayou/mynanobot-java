# 开发流程规范

> **作用域**: 定义 mynanobot-java 从需求到上线的开发流水线、技能调度顺序、阶段门禁、变更追踪。
> **理念**: 人类设计约束 → AI 沿流水线生产 → 机器验证每一步。
> **执行者**: `.harness/agents/owner.md`（Owner Agent）

---

## 1. 流水线总览

```
                                   ┌──────────── 每一步都对照 .harness/rules/ 校验 ────────────┐
                                   │                                                          │
   人类意图                         ▼                                                          │
      │   ① request-analysis  ──▶ ② coding-skill ──▶ ③ unit-test-write ──▶ ④ expert-reviewer ──▶ ⑤ unit-test-ci ──▶ ⑥ deploy-verify ──▶ 交付
      │   规格构建                 测试先行实现       测试完善              专家评审             CI 门禁          部署验证
      ▼
   .harness/changes/<id>/   （全程留档，状态机驱动）
```

每个阶段对应 `.harness/skills/` 下一个技能，**严格按序，不可跳步**。

---

## 2. 阶段定义与门禁

| 阶段 | 技能 | 输入 | 产出 | 出口门禁（Gate） |
|------|------|------|------|-----------------|
| ① 需求分析 | `request-analysis` | 一句话意图 | 规格卡 + 验收标准 + 设计约束 + 测试策略 | 规格明确、AC 可测试、≥3 边界情况、测试策略清晰 |
| ② 编码实现 | `coding-skill` | 已确认的规格卡 | 失败测试 + 最小实现 | 失败测试先存在并转绿、`mvn compile` 通过 + 符合编码规范 |
| ③ 单测编写 | `unit-test-write` | 实现代码 | 边界/降级/回归测试 | 测试通过 + 核心覆盖率 ≥80% + 关键降级路径已覆盖 |
| ④ 专家评审 | `expert-reviewer` | 代码 + 测试 | 评审报告 | 0 个 🔴 严重问题 |
| ⑤ CI 门禁 | `unit-test-ci` | 完整变更 | CI 报告 | 静态分析 + 架构约束 + 全量测试 全绿 |
| ⑥ 部署验证 | `deploy-verify` | 通过 CI 的构建 | 验证报告 | 冒烟 + 健康检查通过，有回滚预案 |

> **门禁原则**: 任一 Gate 不通过 → 退回修复或请示人类，**禁止进入下一阶段**。

---

## 2.1 TDD 小循环（默认适用于核心业务逻辑）

在 ②~③ 阶段内部，核心业务逻辑默认遵循：

```
Red      先写失败测试（基于 AC / 边界 / 降级路径）
Green    写最小实现让测试通过
Refactor 在测试保护下重构
```

适用范围：
- Agent 核心决策逻辑
- 业务规则判断
- 降级 / fallback 路径
- OutputQualityGate

非强制范围：
- 纯 DTO / 配置类 / 一次性脚手架

禁止事项：
- 先写完整实现再回头补核心业务测试
- 用覆盖率代替测试先行

## 3. 变更追踪（.harness/changes/）

每个需求 = 一个变更，分配唯一 ID `C-NNN`，目录结构：

```
.harness/changes/C-NNN-<slug>/
├── change.md          # 变更卡（需求 + 验收标准 + 影响面 + 状态）
├── review.md          # 专家评审记录（④ 阶段产出）
└── verify.md          # 部署验证记录（⑥ 阶段产出，可选）
```

**变更状态机**:

```
draft → analyzing → coding → testing → reviewing → ci → verifying → done
                                                  └─(打回)─┘
```

- 状态写在 `change.md` frontmatter 的 `status:` 字段。
- 状态变更即更新文件 → 天然支持**断点续传**（Owner Agent 读 status 定位当前阶段继续）。

---

## 4. 人机协同协议

### 4.1 人类职责（意图指挥官）
- 定义需求、做架构决策、确定优先级、最终验收（👍/👎）
- 审批 ① 需求卡、④ 评审报告中的关键决策
- 不写实现代码、不写单元测试

### 4.2 Owner Agent 职责（实现执行者）
- 编排技能、编写代码与测试、更新文档、自审计、根因诊断
- 不做架构决策（只给建议）、不改变需求范围

### 4.3 必须暂停请示的触发点
1. 需求模糊到无法确定实现方式
2. 设计方案在实现时被证伪
3. 需要新增依赖 / 改 public API / 改架构
4. 发现安全风险
5. 触碰生产配置或密钥

---

## 5. 特殊场景流程裁剪

| 场景 | 裁剪规则 |
|------|---------|
| **Hotfix（紧急修复）** | 可简化 ①，但必须最小化变更卡（修复描述 + AC），修复后补全；④⑤ 不可省 |
| **Spike（技术探索）** | 可跳过完整 ①②③，但探索结论必须沉淀为 ADR 或 wiki 更新；探索代码不直接合主干 |
| **棕地修改（改已有功能）** | 先更新对应 wiki/changes，再走 ②~⑥ |
| **纯文档变更** | 仅走 ① + ④（评审），无需 ③⑤⑥ |

---

## 5.1 状态流转责任

| 场景 | 责任人 | 动作 |
|------|--------|------|
| 新建变更 | Owner Agent | 创建目录与 `change.md`，状态置为 `analyzing` |
| 需求卡完成待审批 | Owner Agent | 明确提示“请审批”，不擅自进入 `coding` |
| 需求审批通过 | 人类（或经人类明确授权的 Owner Agent） | 将状态从 `analyzing → coding` |
| 编码完成 | Owner Agent | `coding → testing` |
| 测试完成 | Owner Agent | `testing → reviewing` |
| 评审通过 | Owner Agent | `reviewing → ci` |
| CI 全绿 | Owner Agent | `ci → verifying` |
| 验证通过 | Owner Agent | `verifying → done` |
| 评审/CI/验证打回 | Owner Agent | 按问题性质回退到对应上游状态，并在 `change.md` 或 `review.md` 记录原因 |

默认规则：没有人类审批，不得从 `analyzing` 进入 `coding`。

## 6. 迭代节奏

```
单次协作周期（一个变更）:
  ① 读上下文（rules + wiki + change）
  ② 小步实现（每步编译通过）
  ③ 自审计（对照 AC 与 rules）
  ④ 修复 → 报告 → 人审 → 提交 → 更新 change 状态

每日:  完成的变更已提交，change 状态已更新
每周:  熵管理 —— 扫描代码异味、架构漂移、登记技术债
```

---

## 7. 质量博弈（防偷懒 / 防过度设计）

| 信号 | 应对 |
|------|------|
| "差不多了" | 逐条对齐 change.md 中的每个 AC |
| "基本实现了" | 追问：边界情况、错误处理、日志、Javadoc 呢？ |
| "先把代码写出来再补测试" | 退回 ②，要求先基于 AC / 边界写失败测试 |
| 残留 TODO/FIXME | 转为 `.harness/changes/` 任务条目 |
| 引入不必要抽象 | YAGNI —— 要求解释"现在"的价值 |
| 为未来需求写代码 | 砍掉，转为新变更登记 |

---

## 8. 流程合规检查清单（每个变更收口）

- [ ] 走完 ①~⑥ 六个阶段，每个 Gate 通过
- [ ] `.harness/changes/<id>/change.md` 状态 = `done`
- [ ] commit message 含变更 ID
- [ ] 涉及的 `.harness/wiki/` 文档已同步
- [ ] 无残留 TODO/FIXME，无被绕过的约束

---

## 9. 技术债管理（熵管理）

```
登记位置: .harness/changes/ 中专设技术债条目，或汇总于 .harness/changes/TECH-DEBT.md
优先级:
  P0  影响系统稳定性/安全性 —— 一周内修复
  P1  影响开发效率         —— 本月内修复
  P2  代码异味             —— 下个迭代修复
规则: 每个迭代至少处理 1 个 P1+ 技术债
```

**熵检测信号**（出现即登记/告警）: 模块间循环依赖 · ArchUnit 测试被注释/绕过 · Agent 直调另一 Agent 内部方法 · 超 500 行类或 50 行方法 · 超 3 个月未解决的 P1+ 技术债。

**周期任务**: 每周扫码异味 · 每月架构漂移分析（跑全部 ArchUnit）+ 依赖升级评估 · 每季度全面架构审查 + 规则修订。

---

## 9.1 容量与压力门禁（准生产）

当变更进入“准生产/生产级高可用”阶段时，除 ①~⑥ 常规门禁外，至少补充以下验证：
- 主链路并发请求下的成功率
- 单个 Agent 的 P95 延迟与错误率
- Redis / Nacos 抖动时的退化行为
- fallback 命中率是否在预期范围
- 外部依赖限频/超时下是否仍能返回可接受结果

此类验证可作为独立 change，也可并入 `deploy-verify` 的扩展步骤；未做容量验证，不得宣称“生产级高可用”。

---

## 9.2 任务拆解粒度约束

- 单个 task 目标工期应尽量 ≤ 1 天。
- 单个 task 默认只覆盖一个主模块；若跨多个模块，必须说明依赖关系与拆分理由。
- 涉及 `agent + server + tests` 的变更，优先拆成多个 task，避免一把梭大改。
- 一个 change 下可以有多个 task，但 task 间应形成 DAG，无环依赖。

若 task 过大导致无法明确验收、无法单独编译/测试/提交，视为拆解失败，应回退重拆。

## 10. 版本策略（语义化版本）

```
MAJOR.MINOR.PATCH
  MAJOR  Agent 架构变更 / 破坏性 API 变更
  MINOR  新 Agent / 新 Skill / 新功能
  PATCH  Bug 修复 / 性能优化 / 文档更新
```

规则文档（`.harness/rules/`）变更需在 commit message 注明，重大约束调整需人类审批。
