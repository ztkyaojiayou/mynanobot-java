# 运行时可靠性规范

> **作用域**: 约束 mynanobot-java 在准生产/生产环境下的高可用、可观测、可回滚与容量验证要求。
> **定位**: 补充 `工程结构.md / 编码规范.md / 开发流程规范.md`，聚焦“能稳定跑”而不只是“能跑通”。
> **状态**: 强制（生产级交付前必须满足）

---

## 1. 可靠性目标

- 单个外部依赖故障不能直接打死整条规划链路
- A2A 协同链路必须可超时、可重试、可降级、可追踪
- 核心请求失败时必须能定位到 `traceId`
- 生产发布必须有回滚路径

---

## 2. 外部依赖可靠性基线

所有外部依赖（LLM / 小红书 / 地图 / Nacos / Redis）至少满足：
- 明确 timeout
- 明确 retry
- 明确 rate limit
- 明确 fallback
- 明确日志字段
- 明确错误分类

同一依赖的治理策略应尽量统一实现，不允许在多个 Agent 中复制粘贴。

---

## 3. A2A 协同可靠性基线

- `TASK_ASSIGN` 必须带 `traceId`
- 超时必须显式传播，不允许无限等待
- 子 Agent 返回失败时，Supervisor 必须有 fallback 或降级策略
- A2A payload 变更遵循向后兼容优先
- 调用链深度 ≤ 5

---

## 4. Redis / Nacos 退化策略

- Redis 不可用时，允许退化到内存级短暂能力，但必须记录告警日志
- Nacos 抖动或短时不可用时，已注册 Agent 不应立刻整体失效，应有本地缓存/重试恢复策略
- 任何退化都必须可观测，不允许“静默成功”

---

## 5. 可观测性基线

关键链路至少观测：
- Agent 调用延迟
- Agent 调用错误率
- fallback 命中率
- HITL 触发次数
- Redis / Nacos 可用性
- PlanNotebook 步数异常

日志最小字段遵循 `编码规范.md`。

---

## 6. 容量与压测基线

生产级交付前至少回答以下问题：
- 单机可承受多少并发规划请求
- 哪个 Agent 最先成为瓶颈
- Redis / Nacos / 外部 API 哪个先达到上限
- fallback 开启后系统是否仍可维持核心服务

压测结果应记录在独立 change、verify 报告或运维文档中。

---

## 7. 回滚基线

- 每次发布必须记录上一稳定版本
- 每次发布必须有明确回滚命令或回滚步骤
- 不允许依赖不可逆迁移后再宣称可回滚
- `deploy-verify` 未写回滚方案，不得视为生产级通过

---

## 8. 何时可宣称“生产级高可用”

仅当以下条件同时满足时，才可宣称达到生产级高可用：
- 主链路功能通过
- CI 全绿
- 关键链路降级可用
- 可观测性齐全
- 容量/压测有结果
- 回滚路径明确

否则只能宣称“主链路可运行”或“准生产”。
